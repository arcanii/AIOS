## Issues Found and Fixed

### 1. MMU Not Disabled (v0.4.91 crash)

**Symptom:** elfloader memset crashes at low physical address (~0x5000).

**Root cause:** RPi4 firmware leaves MMU enabled with page tables that protect
low physical memory (spin table region). The diagnostic stub disabled D-cache
but not the MMU.

**Fix:** Added full MMU + D-cache + I-cache disable + TLB flush in the
diagnostic stub before jumping to seL4 elfloader:

```asm
dsb sy
mrs x0, sctlr_el2
bic x0, x0, #1        /* M = MMU off */
bic x0, x0, #4        /* C = D-cache off */
bic x0, x0, #0x1000   /* I = I-cache off */
msr sctlr_el2, x0
isb
ic  iallu
tlbi alle2
dsb sy
isb
```

Also added D-cache disable at stub entry (before VideoCore mailbox, which
requires coherent DMA access -- new EEPROM enables D-cache at boot).

### 2. PHYS_BASE Overlap with Elfloader

**Symptom:** `ensure_phys_range_valid` returned -1. Kernel load region
[0x1000, 0x24E000] overlapped elfloader at [0x81000, 0x1DA0A8].

**Root cause:** RPi4 DTS reserves only 4KB (0x0-0x1000) for spin tables.
seL4 build system computes `physBase = 0x1000` (first non-reserved address).
Kernel linked to load at 0x1000, which contains the elfloader.

**Fix:** Changed DTS reservation from 0x1000 to 0x200000 (2MB):
- `deps/kernel/tools/dts/rpi4.dts`: `/memreserve/` 0x1000 -> 0x200000
- `deps/kernel/src/plat/bcm2711/overlay-rpi4.dts`: reserved-memory 0x1000 -> 0x200000

Result: `PHYS_BASE_RAW = 0x200000`, kernel loads above elfloader.

### 3. SMP Boot Hang

**Symptom:** elfloader hangs after `init_boot_vspace()`.

**Root cause:** `KernelMaxNumNodes=4` causes `smp_boot()` to try waking
secondary cores. Cores are in diagnostic stub `wfe` loop without spin table
protocol. Elfloader waits indefinitely for cores to respond.

**Fix:** Set `KernelMaxNumNodes=1` in `settings-rpi4.cmake` (temporary).
SMP needs proper spin table wakeup implementation.

### 4. fb_info Struct Overwritten

**Symptom:** HDMI diagnostic display stops mid-boot.

**Root cause:** Diagnostic stub stores framebuffer info at physical 0x1000.
Kernel memset to 0x1000 (and later root task memset) zeroes this struct,
killing all subsequent HDMI output.

**Fix:** Moved fb_info from 0x1000 to 0x3A000000 (above all loaded images,
below GPU memory at 0x3B400000).

### 5. DTB Memmove Crash (inherited from v0.4.91)

**Symptom:** DTB copy to low physical address crashed.

**Root cause:** Same as issue 1 -- MMU protected low memory.

**Fix:** Fixed by MMU disable. DTB memmove re-enabled in elfloader.

## Boot Chain (working, v0.4.92)

```
RPi4 firmware (start4.elf)
  -> kernel8.img at kernel_address=0x350E000
    -> [0x000-0xFFF] diagnostic stub
       - D-cache disable (SCTLR_EL2.C=0)
       - VideoCore mailbox framebuffer init (1024x768)
       - fb_info stored at phys 0x3A000000
       - Build ID (CRC32) displayed on HDMI
       - MMU + I-cache + D-cache disable, TLB flush
       - Jump to seL4 at stub + 0x1000
    -> [0x1000+] seL4 elfloader
       - load_images: kernel at 0x200000, root task at 0x442000
       - leave_hyp: EL2 -> EL1
       - init_boot_vspace: EL1 page tables
       - arm_enable_mmu: jump to seL4 kernel
  -> seL4 kernel running at EL1
  -> Root task executing (LED on)
```
