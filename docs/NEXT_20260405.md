# AIOS Next Steps — 2026-04-05
## Starting from v0.4.47

## 1. Modularize aios_root.c

### Problem
`src/aios_root.c` has grown to ~2500 lines. It contains boot init, 4 server
functions, fork/exec helpers, reaping logic, and the VFS/driver init. This
makes it hard to navigate, debug, and reason about.

### Proposed Split

```
src/
├── aios_root.c              (~500 lines) Boot, init, main, driver setup
├── servers/
│   ├── exec_server.c        (~350 lines) exec_thread_fn, ELF loading
│   ├── pipe_server.c        (~500 lines) pipe_server_fn, pipe ops
│   ├── thread_server.c      (~200 lines) thread_server_fn, create/join
│   └── tty_server.c         (exists)     TTY line discipline
├── process/
│   ├── fork.c               (~350 lines) do_fork, fork_copy_*, fork_share_*
│   ├── exec.c               (~200 lines) PIPE_EXEC handler logic
│   ├── reap.c               (~100 lines) reap_check, reap_forked_child, zombie table
│   └── wait.c               (~80 lines)  PIPE_WAIT handler, wait_pending table
├── lib/
│   └── aios_posix.c         (exists)     POSIX syscall shim
├── ext2.c                   (exists)     ext2 driver
├── vfs.c                    (exists)     VFS layer
└── procfs.c                 (exists)     /proc filesystem
```

### Shared State

All server threads run in root's VSpace and share these globals. Extract
to `include/aios/root_shared.h`:

```c
/* Root task shared state — all servers access these */
extern vka_t vka;
extern vspace_t vspace;
extern simple_t simple;
extern seL4_CPtr serial_ep_cptr;
extern seL4_CPtr fs_ep_cap;
extern seL4_CPtr thread_ep_cap;
extern seL4_CPtr auth_ep_cap;
extern seL4_CPtr pipe_ep_cap;

/* Process table */
extern active_proc_t active_procs[MAX_ACTIVE_PROCS];

/* File-scope ELF buffer (shared between exec_thread, fork, exec) */
extern char elf_buf[1024 * 1024];

/* ext2 + block driver state */
extern ext2_ctx_t ext2;
extern volatile uint32_t *blk_vio;
extern uint8_t *blk_dma;
extern uint64_t blk_dma_pa;
```

### Migration Strategy

1. Create `include/aios/root_shared.h` with externs
2. Move `active_proc_t`, `elf_seg_info_t`, `wait_pending_t`, `zombie_t` typedefs there
3. Extract `fork.c` first (cleanest boundary — helper functions + do_fork)
4. Extract `reap.c` + `wait.c` (small, depend only on active_procs)
5. Extract `pipe_server.c` (calls fork/exec/reap via function pointers or direct)
6. Extract `exec_server.c` (exec_thread_fn)
7. Extract `thread_server.c` (thread_server_fn)
8. Keep boot/init/driver in `aios_root.c`
9. Update CMakeLists.txt to compile new source files

### Risks

- **Static locals**: exec_thread has `static char prog_name[128]` etc. These
  must stay local to their functions or move to file scope carefully.
- **Thread safety**: Servers run on different TCBs but share globals. Currently
  safe because seL4 cooperative scheduling means no preemption within a server.
  On SMP, servers on different cores CAN run concurrently — but they access
  different proc slots so it works in practice.
- **elf_buf contention**: exec_thread and pipe_server both use elf_buf. They
  run on different threads. If exec happens while fork is running, elf_buf is
  corrupted. This is already a latent bug — modularization doesn't make it
  worse, but we should document it. Fix: per-server buffer or mutex.

### CMakeLists.txt Changes

```cmake
# Add new source files to aios_root target
target_sources(aios_root PRIVATE
    ../../src/servers/exec_server.c
    ../../src/servers/pipe_server.c
    ../../src/servers/thread_server.c
    ../../src/process/fork.c
    ../../src/process/exec.c
    ../../src/process/reap.c
    ../../src/process/wait.c
)
```

## 2. TTY Phase 2: getty/shell Separation

### Current State

mini_shell (1250 lines) handles everything: login prompt, password verification,
MOTD display, command parsing, exec launching, pipe handling. Login is baked into
the shell process.

### Unix Architecture Goal

```
                     ┌──────────────────┐
                     │    aios_root     │
                     │   (init/PID 1)   │
                     └────────┬─────────┘
                              │ fork+exec for each VT
                     ┌────────┴─────────┐
                     │      getty       │
                     │ (per-VT login)   │
                     │                  │
                     │ 1. Print banner  │
                     │ 2. Read username │
                     │ 3. Read password │
                     │ 4. Verify auth   │
                     │ 5. exec(shell)   │
                     └────────┬─────────┘
                              │ exec replaces getty
                     ┌────────┴─────────┐
                     │   mini_shell     │
                     │ (logged-in user) │
                     │                  │
                     │ No login code    │
                     │ Just command     │
                     │ parsing + exec   │
                     └──────────────────┘
```

### Implementation Plan

#### Step 1: Create getty program (`src/apps/getty.c`)

```c
/* getty.c — login prompt for a virtual terminal
 *
 * Displays banner, reads username/password, verifies via AUTH server,
 * then exec's mini_shell with the authenticated UID/GID.
 */
int main(int argc, char *argv[]) {
    /* Display banner */
    printf("\n============================================\n");
    printf("  AIOS 0.4.x\n");
    printf("============================================\n\n");

    /* Login loop */
    while (1) {
        printf("AIOS login: ");
        char username[64];
        fgets(username, sizeof(username), stdin);
        strip_newline(username);

        printf("Password: ");
        /* disable echo */
        char password[64];
        read_password(password, sizeof(password));

        /* Verify via AUTH_LOGIN IPC */
        if (auth_login(username, password) == 0) {
            printf("\nWelcome, %s\n\n", username);
            /* exec mini_shell with authenticated identity */
            execl("/bin/mini_shell", "mini_shell", NULL);
            printf("exec failed\n");
            return 1;
        }
        printf("\nLogin incorrect\n\n");
    }
}
```

#### Step 2: Strip login from mini_shell

Remove from mini_shell:
- Login prompt and password reading (~100 lines)
- Auth verification code (~50 lines)
- MOTD display (~20 lines)
- Banner display (~10 lines)

mini_shell becomes a pure command interpreter that assumes the user is
already authenticated. UID/GID come from the CWD string in argv (set by
exec from getty).

#### Step 3: Root task spawns getty instead of mini_shell

In `aios_root.c`, replace:
```c
proc_add("mini_shell", 200);
```
with:
```c
/* Fork and exec getty for VT0 */
/* (Later: one per virtual terminal) */
proc_add("getty", 200);
```

Actually, for Phase 2 the root task still spawns via exec_thread (not
fork+exec). The getty process is spawned like any other boot program.

#### Step 4: Propagate UID/GID through exec

When getty calls `execl("/bin/mini_shell", ...)`, the exec path (PIPE_EXEC)
must propagate the authenticated UID/GID to the new process. This is
already handled via the CWD string format: `uid:gid:/path`.

#### Step 5: Shell logout → respawn getty

When mini_shell exits (user types `logout` or `exit`), the parent (root
init loop or a simple respawner) detects the exit and spawns a new getty.

### Dependencies

- **fork+exec**: DONE (v0.4.47)
- **waitpid**: DONE (v0.4.46)
- **exec argv passthrough**: NEEDED (currently exec doesn't pass user args)
- **tty_server Phase 1**: DONE (v0.4.44)

### exec argv Passthrough Design

Current PIPE_EXEC only sends the path. For getty→shell, we need to pass
arguments (at minimum, the program name).

Options:
1. **Pack argv after path in MRs**: Path + null + argv strings. Limited to
   ~900 bytes but sufficient for login flow.
2. **Use pipe for argv**: Write argv to a temporary pipe, pass pipe ID in MR.
   More complex but unlimited size.
3. **Encode in CWD string**: Already done for UID/GID. Could extend.

Recommendation: Option 1 (MR packing) for Phase 2. Simple, sufficient.

Format in MRs:
```
MR0..MRn: path\0arg1\0arg2\0\0
```
Double null terminates the argument list.

## 3. TTY Phase 3: Multiple Virtual Terminals (Future)

### Architecture

```
tty_server manages 4 VTs (Ctrl-A 1/2/3/4 to switch):

  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │  VT 0   │  │  VT 1   │  │  VT 2   │  │  VT 3   │
  │ getty/sh │  │ getty/sh │  │ getty/sh │  │ getty/sh │
  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘
       │            │            │            │
  ┌────┴────────────┴────────────┴────────────┴────┐
  │              tty_server                         │
  │  4 line disciplines, 4 input buffers            │
  │  Ctrl-A N switches active VT                    │
  │  Only active VT receives UART input             │
  │  Only active VT's output goes to UART           │
  └─────────────────────────────────────────────────┘
```

Each VT has:
- Its own line discipline (cooked/raw mode, echo state)
- Its own input buffer
- Its own set of processes (getty → shell → user programs)
- Independent login sessions

### Implementation Steps

1. tty_server: add VT array (4 entries), each with line_buf, mode, echo flag
2. tty_server: Ctrl-A handler — switch active_vt index
3. tty_server: route UART input only to active VT's processes
4. tty_server: route output only from active VT to UART
5. Root task: fork+exec 4 gettys, one per VT
6. VT identification: each getty receives VT number in argv

## 4. Other Pending Items

### exec argv passthrough (Priority: HIGH)
Currently exec only sends path. Need to pass argv for:
- getty calling exec("/bin/mini_shell")
- shell calling exec("/bin/ls", "-l", "/etc")
- any program using execvp with arguments

### _exit() exit code capture (Priority: LOW)
musl's _exit() does raw svc, bypassing sel4runtime callback.
Fix: intercept __NR_exit_group in aios_sys_exit_group to send
PIPE_EXIT before the fault.

### pipe_ep visibility (Priority: LOW)
Changed from static to non-static for fork_test extern access.
Should restore to static and remove the extern from fork_test.

### SMP child output ordering (Priority: LOW)
On SMP, parent and child can print simultaneously, causing interleaved
output. Fix: tty_server could buffer per-process output and flush
atomically. Or: accept as correct SMP behavior.

## Priority Order

1. **exec argv passthrough** — blocks TTY Phase 2
2. **Modularize aios_root.c** — improves maintainability for all future work
3. **TTY Phase 2 (getty/shell)** — proper Unix login architecture
4. **TTY Phase 3 (multi-VT)** — multi-user support
5. **_exit() fix** — correctness improvement
6. **COW fork** — performance (v0.5.x)
