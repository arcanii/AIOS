### Elfloader Progress (direct boot with diagnostic stub)

The seL4 elfloader gets through these stages (confirmed by HDMI progress bars):

1. PASS: Enter main()
2. PASS: initialise_devices() returns OK
3. PASS: platform_init() (no-op for BCM2711)
4. PASS: Enter load_images()
5. PASS: cpio_get_file("kernel.elf") -- kernel found in CPIO
6. PASS: elf_checkFile() -- kernel ELF valid
7. PASS: elf_getMemoryBounds() -- bounds OK
8. PASS: DTB found in CPIO, fdt_size OK, range valid
9. PASS: DTB memmove SKIPPED (crashes if enabled -- writes to low phys addr)
10. PASS: Enter load_elf(), enter unpack_elf_to_paddr()
11. **CRASH: memset((void *)dest_paddr, 0, image_size)**

### Root Cause Analysis

The `memset` zeros the kernel destination region before copying ELF segments.
`dest_paddr` is very low (~0x5000, from bar width). `image_size` is ~1MB.

The memset implementation is pure C (no DC ZVA), so it should work. Possible causes:

1. **RPi4 firmware leaves MMU on** with protection for low memory (spin table region).
   The diagnostic stub entry disables D-cache but does NOT disable the MMU.
   If the firmware's stage 1 MMU has low memory mapped read-only, writes fault.

2. **Memory not accessible** at dest_paddr. The seL4 kernel's physical load address
   (from its ELF LOAD segment) targets very low physical RAM. The RPi4 DTS reserves
   the first page (0x0-0xFFF) for spin tables. If the kernel loads below that...

3. **DTB memmove has the same issue**: it also writes to `ROUND_UP(kernel_phys_end, PAGE_BITS)`
   which is a low address. Both the DTB copy and kernel zeroing crash at the same type
   of operation: large writes to low physical memory.

### Likely Fix

The diagnostic stub needs to **disable the MMU** (not just D-cache) before jumping
to the seL4 elfloader. On AArch64 EL2:

```asm
mrs x0, sctlr_el2
bic x0, x0, #1        /* clear M bit = MMU off */
bic x0, x0, #(1<<2)   /* clear C bit = D-cache off */
msr sctlr_el2, x0
isb
```

This should be tried first in the next session.
