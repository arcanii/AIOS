# AIOS Next Steps -- 2026-04-10b

Starting from **v0.4.74**.

## Completed in v0.4.74

### M1: virtio-net device detection + driver/server skeleton

- boot_fs_init.c scans virtio MMIO range, identifies virtio-net (device_id=1)
- boot_net_init.c allocates 128KB DMA region (32 pages via single untyped Retype)
- Two-thread architecture: net_driver (RX/TX virtqueue management) + net_server (protocol dispatch)
- Phase 1 ISR polling (no IRQ notification yet)
- UART batch drain to avoid serial contention during network I/O

### M2: ARP + ICMP ping

- Protocol stack implemented in net_stack.c
- ARP cache (8 entries, timeout-based expiry)
- ICMP echo request/reply (ping)
- Gratuitous ARP on startup to announce presence on the network

### M3: POSIX socket API + UDP

- Socket syscalls added: 198 (socket), 200 (bind), 201 (listen), 202 (accept4), 206 (sendto), 207 (recvfrom)
- net_ep (network endpoint capability) propagated to child processes via exec_server + pipe_server
- Socket table: 8 slots per process
- UDP send and receive working end-to-end

### M4: TCP state machine + HTTP server

- handle_tcp ported from v0.3.x codebase
- TCP states implemented: LISTEN, SYN_RCVD, ESTABLISHED, FIN_WAIT
- httpd.c user program serves static HTTP responses
- curl localhost:8080 works from host through QEMU port forwarding

## Bugs Found and Fixed

- **Ethernet padding inflates TCP data length**: Minimum 60-byte Ethernet frames include padding that was incorrectly counted as TCP payload. Fixed by using IP total length field instead of Ethernet frame length.
- **RX kick before DRIVER_OK**: Virtqueue kick was issued before the DRIVER_OK status bit was set, causing the device to ignore the first batch of RX buffers. Moved kick after status write.
- **pipe_server PIPE_EXEC missing net_ep**: The PIPE_EXEC path in pipe_server did not propagate the network endpoint to forked+exec children. Added net_ep to the capability set copied during PIPE_EXEC.
- **accept4 syscall 202 vs 242**: musl uses syscall 242 for accept4 on AArch64, but the initial implementation used 202 (accept). Added handler for both numbers.
- **Ctrl-C force-kill double-free**: Force-kill path could double-free the process structure if the first Ctrl-C had already begun teardown. Added a killed flag to prevent re-entry.

## Lessons Learned

- **Notification bind works but wakeup unreliable on SMP**: seL4 notification wakeup does not always wake the target thread on multi-core. Polling is more reliable for now.
- **Kick after DRIVER_OK**: virtio spec requires the device to be in DRIVER_OK state before any virtqueue operations. Kicking earlier is silently ignored.
- **Ethernet minimum 60 bytes**: Frames shorter than 60 bytes are padded by the NIC/hypervisor. The padding must not be counted as protocol payload.
- **Two exec paths need same caps**: Both the direct exec path (exec_server) and the fork+exec path (pipe_server PIPE_EXEC) must propagate the same set of endpoint capabilities, including net_ep.
- **accept vs accept4 numbering**: AArch64 Linux does not have a separate accept syscall number. musl always uses accept4 (242). The generic accept (202) is only used on older architectures.

## Bugs Still Open

- **virtio-blk stale reads**: Partially mitigated by dmb sy barrier (v0.4.73), needs stress testing
- **Signals during blocking syscalls**: Signal delivery while a process is blocked in a syscall (e.g., read, recvfrom) does not interrupt the call
- **Notification wakeup**: SMP notification wakeup unreliable, currently using polling as workaround

## QEMU Command (with networking)

```
qemu-system-aarch64 \
    -machine virt,virtualization=on \
    -cpu cortex-a53 -smp 4 -m 2G \
    -nographic -serial mon:stdio \
    -drive file=disk/disk_ext2.img,format=raw,if=none,id=hd0 \
    -device virtio-blk-device,drive=hd0 \
    -drive file=disk/log_ext2.img,format=raw,if=none,id=hd1 \
    -device virtio-blk-device,drive=hd1 \
    -device virtio-net-device,netdev=net0 \
    -netdev user,id=net0,hostfwd=tcp::8080-:80 \
    -kernel build-04/images/aios_root-image-arm-qemu-arm-virt
```

The `-netdev user,id=net0,hostfwd=tcp::8080-:80` flag forwards host port 8080 to guest port 80, allowing `curl localhost:8080` from the host to reach the AIOS HTTP server.
