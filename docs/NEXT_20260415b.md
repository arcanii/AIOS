# AIOS Next Steps -- 2026-04-15b

Continuing **v0.4.95** TCC self-hosting investigation.

## Session Result: TCC Self-Hosting Proven (Compilation)

All 12 TCC source files compile natively inside AIOS using the AIOS-native
TCC compiler. The link step requires host-assisted cross-linking due to
memory constraints. A host-linked tcc2 binary runs on AIOS and compiles
programs successfully.

## Features Implemented

### 1. TCC Individual Source Compilation (all 12 files)

Each TCC source file compiles successfully with:
```
tcc -c -DONE_SOURCE=0 -D_Thread_local= -DTCC_TARGET_ARM64 -DCONFIG_TCC_STATIC \
    -I /usr/src/tcc -I /usr/src/tcc/include -nostdinc -I /usr/include \
    /usr/src/tcc/<file>.c -o /tmp/<file>.o
```

Files verified: libtcc.c, tccpp.c, tccgen.c (265 KB, largest), tccdbg.c,
tccelf.c, tccasm.c, tccrun.c, tcctools.c, arm64-gen.c, arm64-link.c,
arm64-asm.c, tcc.c.

### 2. Key Build Flag Discoveries

| Flag | Purpose |
| --- | --- |
| -DONE_SOURCE=0 | Prevents libtcc.c from #including all other .c files |
| -D_Thread_local= | Blanks out _Thread_local in musl errno.h (TCC lacks TLS) |
| -nostdinc -I /usr/include | Explicit include path (avoids TCC built-in paths) |

Without -DONE_SOURCE=0, libtcc.c defaults to ONE_SOURCE=1 and includes
everything, creating a single massive compilation unit that exceeds memory.

### 3. sym_test.c: Anonymous Union Struct Test

Created test program mimicking the TCC Sym struct layout (4 anonymous
unions, cleanup_label in the 4th). Result: 7/7 PASS. Proves TCC handles
the exact pattern that caused the "field not found: cleanup_la" error in
v0.4.94. The error was caused by missing flags (not a TCC limitation).

### 4. Archive Deduplication Analysis

MRI-merged libaios_full.a has 1418 members with 17 duplicate names
(clone.o, syscall.o, elf.c.obj, etc. from different source archives).
Findings:
- Duplicate names cause TCC linker to freeze (infinite scanning loop)
- Python ar parser dedup works (extracts all members, renames duplicates)
- However, ar x loses within-archive duplicates (broke _Exit/__syscall)
- MRI merge reverted as default (GNU ld handles duplicates fine)

### 5. libc_min.a Concept

Host-side link analysis with aarch64-linux-gnu-ld --Map identifies which
libaios_full.a members are actually needed by TCC. Result: ~578 KB vs
3.9 MB full archive. This approach needs refinement but proves the concept
of reducing link-time memory pressure.

### 6. tcc2 Binary (Host-Assisted Link)

Cross-compiled TCC .o files (aarch64-linux-gnu-gcc -c -DONE_SOURCE=0)
combined via relocatable link (ld -r) into tcc_all.o (519 KB), then
linked with aios-cc on the host. Resulting tcc2 (670 KB) runs on AIOS:
```
/tmp/tcc2 -o /tmp/hello2 /tmp/hello.c
/tmp/hello2   -> "hello from tcc"
```

## Infrastructure Blockers Identified

### Blocker 1: VKA Resource Leak on Process Exit

**Problem:** fork+exec permanently consumes VKA pages (~3072 for 12 MB
morecore BSS + page tables + caps). Resources are NOT reclaimed when the
process exits. After ~2 fork+exec operations, the 8000-page VKA pool is
exhausted and subsequent forks fail.

**Impact:** Cannot run build_tcc.sh (needs 12 compilations + 1 link =
13 fork+exec operations). Even sourced script (. /tmp/build_tcc.sh)
fails after 2 compilations.

**Root cause:** reap.c and pipe_server.c cleanup paths do not call
vka_free_object for allocated frames. seL4 allocman requires explicit
CNode_Revoke + CNode_Delete + vka_free_object to return resources.

**Fix approach:** Track all VKA allocations per-process during fork/exec.
On process exit, iterate tracked allocations and free each one. Requires:
- Per-process allocation list in active_proc_t or similar
- Frame tracking: morecore pages, ELF segment pages, page tables
- Cap tracking: endpoints, TCB, VSpace, CNode entries
- Cleanup in reap_forked_child and PIPE_EXIT handler

### Blocker 2: Fixed Morecore (12 MB, Eagerly Mapped)

**Problem:** The morecore region is a static BSS array in musl
sys_morecore.c. The seL4 ELF loader maps ALL BSS pages at process
creation time, even if the process never touches them. With 12 MB
morecore, each process immediately consumes 3072 VKA pages.

**Impact:** TCC linker needs ~15-20 MB to process tcc_all.o (519 KB) +
libc.a (3.9 MB). But 16 MB morecore (4096 pages/process) + 10000-page
pool fails to boot (too many untypeds consumed by BSS).

**Fix approach:** Implement real mmap MAP_ANONYMOUS:
- Process starts with small morecore (e.g., 2 MB)
- When more memory needed, musl calls mmap(MAP_ANONYMOUS)
- AIOS allocates VKA frames on demand and maps into child VSpace
- Track mapped regions for munmap and exit cleanup
- Benefits: decouples startup cost from maximum heap size

### Blocker 3: Slow Archive I/O (IPC-Based)

**Problem:** Reading 3.9 MB libc.a through AIOS VFS requires ~4333 IPC
round trips (900 bytes per MR-based read). TCC linker makes multiple
passes over the archive for symbol resolution. Combined latency causes
apparent freeze (minutes for complex links).

**Impact:** Even with VKA fix and mmap, linking against full libc.a
would be very slow.

**Fix approach:** SHM-based file read for large reads:
- Map shared frame between user process and root task
- Bulk copy file data to SHM (4096 bytes per transfer vs 900)
- User process reads from mapped SHM buffer
- 4-5x throughput improvement for large file reads

## Next Session Priorities

### Priority 1: VKA Resource Reclamation

This is the single most impactful fix. It enables:
- Multi-step builds (compile all files in one session)
- Repeated program execution without VKA exhaustion
- General system stability for long-running sessions

Implementation plan:
1. Add per-process allocation tracking to active_proc_t
2. Track frame allocations during fork (do_fork) and exec (exec_server)
3. Free all tracked allocations in reap_forked_child
4. Test: run 10+ programs sequentially without VKA exhaustion

### Priority 2: Real mmap MAP_ANONYMOUS

Enables dynamic memory growth beyond fixed morecore:
1. Add mmap/munmap syscall handlers in posix_misc.c or new posix_mmap.c
2. Allocate VKA frames on demand, map into child VSpace
3. Track mapped regions per-process for cleanup
4. Reduce default morecore to 2-4 MB (processes grow as needed)
5. Test: TCC link step with libc.a succeeds natively

Improves large file read performance:
1. Extend FS_PREAD to support SHM transfer mode
2. Map shared frame in posix_file.c for reads > threshold
3. Bulk copy in fs_server, user reads from SHM
4. Test: TCC archive scanning completes in seconds not minutes

## Key Learnings

### ONE_SOURCE Default

TCC libtcc.c defaults to ONE_SOURCE=1 if not explicitly set:
```c
#ifndef ONE_SOURCE
# define ONE_SOURCE 1
#endif
```
Multi-file builds MUST pass -DONE_SOURCE=0 to prevent libtcc.c from
including all other .c files.

### _Thread_local in musl errno.h

musl errno.h:16 has `extern _Thread_local int errno;`. TCC does not
implement _Thread_local. Fix: -D_Thread_local= (define as empty).
Safe for AIOS single-threaded user processes.

### Morecore Pages Are Eager

seL4 ELF loader maps ALL BSS pages at process creation. A 12 MB
morecore = 3072 pages consumed immediately, even if brk is never called.
This is the primary VKA consumer and the reason processes are so expensive.

### Three-Process Problem

`dash /tmp/build_tcc.sh` creates 3 processes simultaneously:
login-dash + script-dash + tcc = 3 x 3072 = 9216 pages > 8000 pool.
Workaround: source the script (`. /tmp/build_tcc.sh`) to avoid the
extra dash process. Even then, VKA leak limits to ~2 compilations.

### Archive Duplicate Members

MRI merge produces archives with duplicate member names when source
archives contain identically-named objects (clone.o in musl, elf.c.obj
in libelf and libsel4utils). GNU ld handles this correctly. TCC linker
freezes on duplicate members (scanning loop). Python ar parser dedup
resolves this but ar x loses within-archive duplicates.
