# AIOS Next Steps -- 2026-04-07

## Starting from v0.4.50

## Completed in Session 2026-04-06 (Part 3)

### Notification-Based Fault Delivery
- Child fault_ep is now a minted badged pipe_ep (not a separate endpoint)
- Faults arrive on pipe_ep with label < PIPE_CREATE (60), identifiable by badge
- Eliminates destructive seL4_NBRecv, spin loops, and SMP race conditions
- pipe_server handles faults as normal IPC messages in its main loop
- reap.c and fork.c updated for minted fault_ep cleanup

### Fork+Exec Pipe Operator (POSIX)
- Pipe operator converted from legacy EXEC_RUN IPC to fork+pipe+dup2+exec
- run_pipeline() in mini_shell uses pure POSIX: pipe(), fork(), dup2(), execv()
- Blocking PIPE_READ via SaveCaller when pipe empty + writer alive
- PIPE_CLOSE_WRITE signals EOF to blocked readers
- aios_exit_cb sends PIPE_CLOSE_WRITE before PIPE_EXIT
- Server-side pipe buffer cleanup after pipeline completes
- Works: echo|cat, ls|cat, ls|head, echo|cat|cat (3-stage), repeated use

### Key Bug Fixes
- pipe2() fd allocation: mark first fd active before allocating second
  (both fds got same index -- root cause of ALL prior pipe failures)
- close() must not send PIPE_CLOSE_WRITE (kills pipe before exec runs)
- close() must not send PIPE_CLOSE (destroys buffer before reader sees data)
- MAX_ACTIVE_PROCS increased from 8 to 16
- SMP yield between forks to serialize child startup
- pipe_test diagnostic tool added (4 POSIX tests, all pass)
- debug builtin shows active_procs, pipes, wait_pending state

## Priority 1: SMP Fork Serialization (Replace Yield)

Current workaround: 20x seL4_Yield() between forks in run_pipeline.
Proper fix: PIPE_FORK/PIPE_EXEC completion handshake -- pipe_server
should not accept next IPC until child process has completed exec
and is ready to receive IPC itself. Options:
- Notification from child to parent after exec completes
- pipe_server queues fork requests and processes them sequentially
- Barrier in do_fork that waits for child TCB to reach entry point

## Priority 2: Pipe Buffer Leak from pipe_test

close() on pipe fds no longer frees server-side buffer.
This is correct for pipeline use (parent cleans up via PIPE_CLOSE).
But standalone pipe() users (pipe_test) leak server-side buffers.
Fix: reference-count pipe endpoints, free when both ends closed.

## Priority 3: Getty/Shell Separation

- root spawns getty (CPIO, 5 caps)
- getty: banner + login loop, EXEC_RUN mini_shell
- mini_shell: pure command interpreter (no login code)
- exit/logout returns to getty

## Priority 4: Remove exec_ep from mini_shell

After pipe conversion, mini_shell only uses exec_ep for:
- do_exec_piped (legacy, can be removed)
- do_exec_ipc / do_exec_bg_ipc (used by redirect and background)
Convert redirect to fork+dup2+exec. Remove exec_ep dependency.

## Priority 5: Signal Infrastructure

Required for bash/zsh job control:
- SIGCHLD, SIGINT, SIGTERM
- signal()/sigaction() in aios_posix

## Priority 6: Shared-Memory Pipes (v0.5.x)

Current IPC-based pipes require pipe_server round-trip per chunk.
Shared-memory ring buffers would allow direct writer-to-reader transfer.
Not blocking for current functionality but needed for performance.
