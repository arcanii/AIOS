# AIOS Backlog -- 2026-04-06a

Updated after v0.4.55 commit.

## Critical

### 1. ext2 directory cache: write-then-read fails

ext2_create_file() writes a new directory entry and inode, but
ext2_lookup() cannot find the entry in the same session. The
directory block is re-read from disk but does not reflect the
write. This affects any create-then-open workflow: cp verify,
posix_verify open-after-create, tee+cat, shell redirection
to new files.

Root cause: ext2_write_block() updates the virtio-blk device,
but ext2_lookup() re-reads the directory blocks and may get
stale data if the virtio-blk read path has caching or if the
write did not flush to the block device synchronously.

Fix options:
  a) Verify ext2_write_block flushes synchronously (check virtio
     VIRTQ_DESC_F_WRITE ordering and completion).
  b) Add a small directory-block write cache: after writing a
     directory block, keep it in memory so lookup hits cache first.
  c) Re-read the parent directory inode after create to pick up
     updated size/block pointers.

Impact: HIGH -- blocks posix_verify 42/42, cp workflows, any
program that creates and immediately reads a file.
Files: src/ext2.c, src/vfs.c
Status: NEW (discovered in v0.4.55 posix_verify testing)

### 2. Getty fork failure = dead system

getty.c: if fork() fails, it prints error and loops to login.
Repeated failure (proc table full) means user logs in, sees
error, loops forever.
Fix: retry with backoff (3 attempts, 500ms sleep). On persistent
failure, fall back to running mini_shell directly in getty process
(degraded single-process mode rather than dead system).
Files: src/apps/getty.c
Status: OPEN (from v0.4.53)

## High

### 3. run_pipeline: no cleanup on mid-pipeline fork failure

If fork fails for segment N of M, earlier children are already
running and writing to pipes. Code sets child_pids[si] = -1
and continues, but earlier children may block forever writing
to a pipe whose reader was never spawned.
Fix: on any fork failure, kill+reap already-forked children,
close all pipe fds, destroy server-side pipe buffers. Return
-1 immediately.
Files: src/apps/mini_shell.c (run_pipeline)
Status: OPEN (from v0.4.53)

### 4. Signal Phase 3: cross-process delivery on blocking syscalls

aios_sig_check() only runs on kill/tgkill return paths. For
cross-process delivery (parent sends SIGUSR1 to child), the child
must check sig_pending on return from blocking syscalls: read,
write, nanosleep, waitpid.
Fix: insert aios_sig_check() at end of aios_sys_read, aios_sys_write,
aios_sys_nanosleep, and waitpid wrapper. Return -EINTR if signal
was delivered.
Files: src/lib/aios_posix.c
Status: OPEN (from v0.4.54)

### 5. PIPE_EXEC_WAIT deadlock recovery

If child crashes between PIPE_FORK and PIPE_EXEC (during ELF
load), exec_done[ci] is never set, check_exec_wait is never
called, and the parent shell blocks forever in PIPE_EXEC_WAIT.
Fix: handle_child_fault should call check_exec_wait for the
faulted child index. Alternatively, add timeout via seL4
notification + timer.
Files: src/servers/pipe_server.c
Status: OPEN (from v0.4.53, partially addressed in v0.4.54)

## Medium

### 6. Static buffers in do_exec / do_exec_bg

do_exec() uses static char argv0[64] and argbuf[512].
do_exec_bg() uses static char bg_argv0[64] and bg_argbuf[512].
These would corrupt under concurrent or reentrant calling.
Not a bug today (single-threaded shell), but will bite when
signals or job control are added.
Fix: move all static buffers to stack-local.
Files: src/apps/mini_shell.c
Status: OPEN (from v0.4.53)

### 7. Getty: no fault/signal logging for child shell

waitpid() returns but we only check WIFEXITED. If shell dies
from a fault (WIFSIGNALED), getty silently loops back to login
with no diagnostic.
Fix: check WIFSIGNALED(status) and log signal number.
Files: src/apps/getty.c
Status: OPEN (from v0.4.53)

### 8. getconf: last unported sbase tool

getconf.c requires a generated getconf.h header with system
configuration values (PAGE_SIZE, NGROUPS_MAX, PATH_MAX, etc.).
Write scripts/gen_getconf.py to produce the header, then build
with aios-cc. Would bring sbase to 99/99.
Files: scripts/gen_getconf.py (new), sbase/getconf.c
Status: NEW

## Low

### 9. Shell globbing and quoting

mini_shell does not support glob patterns (*.c, /bin/ch*).
Single and double quoting is minimal (strip outer quotes only).
Proper quoting would allow: echo "hello world" as one arg.
Files: src/apps/mini_shell.c
Status: NEW

### 10. Shell redirection (> and >>)

No file redirection support. Commands like echo hello > file.txt
are parsed as arguments rather than redirect operators.
Requires: ext2 write working correctly (blocked by item 1).
Files: src/apps/mini_shell.c
Status: NEW (blocked by ext2 dir cache)

## Future (v0.5.x)

### 11. Shared-memory pipes

Current IPC pipes: pipe_server round-trip per chunk (~900 bytes).
Shared-memory: vka_alloc_frame mapped into both writer and reader
VSpaces. Ring buffer with atomic head/tail. Zero IPC for data transfer.
pipe_server only handles lifecycle (create/destroy).

### 12. Dynamic ELF buffer

elf_buf is static 1MB. Large programs (gcc, etc.) will not load.
Fix: vka_alloc_frame for variable-size ELF loading.

### 13. Networking: virtio-net + TCP/IP

virtio-net driver + lightweight TCP/IP stack (lwIP or custom).
Would enable tftp, wget, ssh-like functionality.

### 14. ext2 write improvements

Multi-block file write, triple indirect blocks, file append,
truncate, proper block deallocation on unlink.
