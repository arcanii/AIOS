## Completed in v0.4.87

### Signal delivery to blocked PIPE_READ

- **caller_pid tracking in pipe_read_blocked_t**
  - New `caller_pid` field records which process is blocked in PIPE_READ/PIPE_READ_SHM
  - Populated from IPC badge when SaveCaller blocks the reader
  - Enables targeted signal wakeup by PID

- **PIPE_SIGNAL wakes blocked readers with EINTR**
  - `wake_blocked_reader_signal(target_pid)` helper: finds blocked reader by PID, replies with -4 (EINTR sentinel)
  - Called from PIPE_SIGNAL handler after setting `sig_pending` for non-fatal signals (1-31, excluding SIGKILL/SIGTERM)
  - Client-side: `got == -4` dispatches signal via `aios_sig_check()` before returning -EINTR
  - For SIG_DFL handlers, process exits within `aios_sig_check()` (never returns -EINTR)
  - For custom handlers (e.g., dash SIGINT), handler runs via longjmp

- **First Ctrl-C routed through PIPE_SIGNAL**
  - Root task UART handler no longer sets `sig_pending` directly
  - Both first ^C (SIGINT) and second ^C (SIGKILL) go through PIPE_SIGNAL IPC
  - Ensures pipe_server can wake blocked readers on signal delivery
  - Required because direct `sig_pending` manipulation bypasses the wakeup mechanism

- **kill(0, sig) resolves to fg_pid**
  - PIPE_SIGNAL handler: `if (target_pid == 0 && fg_pid > 0) target_pid = fg_pid`
  - Enables SSH relay `kill(0, 2)` to target the foreground child (e.g., cat) without knowing its PID
  - POSIX-compliant: pid 0 targets the process group (approximated via fg_pid)

### SSH Ctrl-C forwarding

- **Signal-only Ctrl-C (no raw byte injection)**
  - `process_input()` Ctrl-C handler: calls `kill(0, 2)` then echoes "^C\r\n" to SSH client
  - Removed raw 0x03 + '\n' byte write to stdin pipe (stale bytes confused dash after child exit)
  - Signal delivery via PIPE_SIGNAL is sufficient -- EINTR wakeup handles blocked reads

- **Stale sig_pending prevention**
  - PIPE_EXEC clears `sig_pending = 0` on new process setup
  - Prevents recycled active_proc slots from inheriting stale signals from previous occupant
  - Root cause: cat killed by SIGINT left sig_pending bit set; next process in same slot dispatched stale SIGINT immediately

### SSH exit and pipe cleanup

- **stdout_wr_fd deferred close**
  - sshd relay closes parent write end of stdout pipe after first shell output (not at relay start)
  - Avoids race with fork+exec: closing before PIPE_EXEC would set write_closed=1 prematurely
  - After close, only dash holds write_ref; dash exit triggers pipe EOF

- **Server-side pipe close in handle_child_fault**
  - Removed `exit_cb_ran` guard: handle_child_fault always closes pipes using server-side `stdout_pipe_id`
  - Root cause: client-side `stdout_pipe_id` is unreliable (dash may reset it during terminal save/restore)
  - Server-side `ap->stdout_pipe_id` from PIPE_EXEC is always correct
  - Guard changed to `write_refs > 0` to prevent negative ref counts

- **PIPE_EXIT no longer clears fg_pid**
  - Removed `fg_pid = -1` from PIPE_EXIT handler
  - fg_pid management handled exclusively by handle_child_fault (promotion to parent)
  - Previous behavior left fg_pid=-1 after clean exit, making processes unkillable via Ctrl-C

- **Exit syscalls close pipes**
  - `aios_sys_exit` and `aios_sys_exit_group` now send PIPE_CLOSE_WRITE/PIPE_CLOSE_READ before PIPE_EXIT
  - Provides client-side pipe close for processes that exit via _exit() syscall path
  - Note: dash exit via sel4runtime callback bypasses these syscalls; server-side close in handle_child_fault is the reliable fallback

### TCP retransmit tolerance

- **Accept retransmissions without data duplication**
  - `seq < rcv_nxt`: retransmission; re-ACK with current rcv_nxt, do not buffer
  - Overlapping retransmissions: if segment tail extends past rcv_nxt, deliver only new bytes
  - `seq > rcv_nxt`: out-of-order; drop and send duplicate ACK
  - Fixes SSH first-connection failure caused by SLIRP retransmitting during key exchange

- **Connection socket owner_pid inheritance**
  - SYN handler sets `conn->owner_pid = sockets[listen_idx].owner_pid`
  - NET_CLEANUP_PID now properly frees connection sockets on process death
  - Previously connection sockets had owner_pid=0, surviving cleanup

### Process lifecycle

- **exit_cb_ran flag in active_proc_t**
  - Set by PIPE_EXIT handler when aios_exit_cb has already run
  - Used to prevent premature fg_pid clearing in PIPE_EXIT
  - Note: not used as pipe close guard (server-side close is authoritative)

## Known Issues

### sshd restart second connection sometimes fails

- First connection after sshd restart may fail with "packet write failed"
- Likely stale TCP connection socket from previous session (same SLIRP source port)
- OpenSSH client retries automatically; second attempt succeeds
- Proper fix: TCP TIME_WAIT state or explicit socket table flush

### enc-write: data failed on clean exit

- Cosmetic: sshd tries to send CHANNEL_EOF/CHANNEL_CLOSE after SSH client has already disconnected
- The TCP socket write fails because the client closed first
- Does not affect functionality (session was already ending)

### SSH Ctrl-C shows extra prompts

- Each ^C shows `# #` (one from ^C echo, one from dash reprompting)
- Cosmetic: functionality is correct

### SSH Ctrl-C requires extra keypress to show prompt

- After ^C kills cat, the new prompt may not appear until the user presses another key
- Cause: sshd relay polls stdout pipe every 10ms; dash prompt arrives on the next poll cycle
- The interrupt IS delivered immediately (cat exits), just the prompt display is delayed
- Fix: reduce poll interval or use notification-based wakeup for pipe data arrival

### sshd must be rebuilt after POSIX shim changes

- sshd is built via aios-cc (not ninja), same as dash
- ninja rebuilds libaios_posix.a but does NOT rebuild sshd
- Must manually rebuild sshd after any change to src/lib/posix_*.c or src/lib/aios_posix.c
- Root cause of several debugging rounds in v0.4.87: posix_file.c close() changes not picked up by sshd
