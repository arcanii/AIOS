# AIOS Next Steps -- 2026-04-08d

## Starting from v0.4.68

## Completed in v0.4.68

* **Dash is now the primary login shell.** Getty reads pw_shell from
  /etc/passwd via getpwuid() and execs the configured shell. The
  /etc/passwd shell field is set to /bin/dash for both root and user
  accounts. mini_shell remains on disk as a fallback but is no longer
  the default.
  Files: src/apps/getty.c, disk/rootfs/etc/passwd

* **TTY echo enabled by default.** tty_server now starts with tty_echo=1.
  All programs see typed characters echoed by the tty line discipline.
  mini_shell and getty switch to RAW mode during their own line editing
  (where they handle echo manually) and back to COOKED mode after.
  Files: src/apps/tty_server.c, src/apps/mini_shell.c, src/apps/getty.c

* **POSIX stdin reads from cooked line queue.** posix_file.c read(0,...)
  now uses TTY_READ (label 71) which reads from the tty_server line
  queue -- completed, line-edited input. Previously used SER_GETC which
  returned raw keystrokes including backspace/DEL bytes. Dash interactive
  mode now receives clean input with proper backspace handling.
  Files: src/lib/posix_file.c

* **Raw/cooked mode buffer isolation.** In RAW mode, characters go only
  to key_buf (for SER_GETC users like mini_shell). In COOKED mode,
  completed lines go to both line_queue (for TTY_READ) and key_buf (for
  legacy SER_GETC). Previously RAW mode leaked raw chars into line_queue,
  causing stale data when switching to a cooked-mode program (dash).
  Both buffers are flushed on mode switch to prevent cross-contamination.
  Files: src/apps/tty_server.c

* **Dash variable expansion works.** for loops, variable assignment, and
  arithmetic expansion all work correctly in dash interactive mode and
  script files. The previous "empty $i" bug was caused by mini_shell
  pre-expanding $VAR before dash saw the command. With dash as the
  primary shell, this is no longer an issue.

* **bump-patch.sh and bump-minor.sh fixed for macOS.** sed -i requires
  an empty string argument on macOS (sed -i '' ...). Both scripts
  updated.
  Files: scripts/bump-patch.sh, scripts/bump-minor.sh

## Lessons Learned in v0.4.68

* **Two input paths in tty_server must be kept separate.** key_buf
  (SER_GETC, raw) is for programs that do their own line editing
  (mini_shell). line_queue (TTY_READ, cooked) is for programs that
  expect terminal-processed input (dash, cat, sbase tools). Mixing
  them -- particularly pushing raw chars to line_queue in RAW mode --
  causes stale data bugs that are hard to diagnose (wrong command
  replayed, garbled input, first-command failures).

* **Mode switches must flush both buffers.** When switching from COOKED
  to RAW (mini_shell starting read_line), the line_queue may have
  cooked data from a previous program. When switching from RAW to
  COOKED (dash starting after mini_shell), key_buf may have raw chars
  from the line editor. Flushing both on every mode switch prevents
  cross-contamination.

* **Dash rebuild after POSIX shim changes.** dash links against
  libaios_posix.a. ninja rebuilds the library but does NOT rebuild
  dash (it is cross-compiled separately via aios-cc). Always rebuild
  dash after changes to src/lib/posix_*.c files.

* **expand_vars in mini_shell ignores quotes.** mini_shell expands
  $VAR references even inside single quotes, which is incorrect shell
  behavior. This caused dash -c 'echo $x' to receive an empty string.
  The fix was to switch to dash as the primary shell rather than fixing
  mini_shell.

* **tty_server echo and line discipline are coupled.** Setting tty_echo=1
  alone is not enough -- the line discipline must also correctly route
  characters to the right buffer. The echo flag controls display; the
  mode (RAW/COOKED) controls routing.
