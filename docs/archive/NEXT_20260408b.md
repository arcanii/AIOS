## Completed in v0.4.66

* **Dash stdout working.** Two root causes fixed:
  1. aios-cc compile step did not pass EXTRA flags (-include config.h,
     -I, -DSHELL) to per-source compilation -- only to the linker.
     config.h defines stat64->stat etc. Without it, dash failed to
     compile (stat64 unknown). Fix: add "${EXTRA[@]}" to compile line.
  2. mini_shell split args by spaces without quote awareness.
     `dash -c "echo hello"` reached dash as 4 argv entries instead of 3.
     New `split_args_qa()` helper respects "..." and single-quote
     delimiters. Applied to all 5 exec paths (do_exec, do_exec_bg,
     redirect-output child, redirect-input child, pipeline child).
     Early strip_quotes loop removed; cd builtin gets targeted strip.
  Files: scripts/aios-cc, src/apps/mini_shell.c

* **FS_APPEND IPC op (label 19).** Server-side append: fs_server reads
  existing file via vfs_read, appends new data, writes combined via
  vfs_create. Same MR layout as FS_WRITE_FILE. posix_file.c write()
  sends FS_APPEND when is_append flag is set on the fd.
  Files: include/aios/ext2.h, src/servers/fs_server.c,
  src/lib/posix_file.c

* **Shell >> workaround removed.** The v0.4.65 read-modify-write hack
  (shell reads file via FS_CAT, prepends to buffer, writes combined
  via FS_WRITE_FILE) is replaced by a single FS_APPEND IPC call.
  mini_shell.c redirect path uses label 19 when redir_append is set.
  Files: src/apps/mini_shell.c

* **PIPE_BUF_SIZE fix.** Changed from 8192 to 4096 to match the shm
  page size. The old value caused a latent out-of-bounds bug: the ring
  buffer wrapped at 8192 but the shm frame was only 4096 bytes.
  Files: include/aios/root_shared.h

* **SHM pipes phase 2 -- server-side infrastructure.** New IPC ops:
  PIPE_MAP_SHM (78), PIPE_WRITE_SHM (79), PIPE_READ_SHM (80).
  Transfer page (xfer) allocated lazily on first PIPE_MAP_SHM request.
  Server copies between xfer page and ring buffer. Blocked reader wake
  function supports SHM flag (is_shm) for correct reply format.
  Client-side mapping DISABLED: sel4utils_destroy_process does not
  reclaim page table objects allocated by post-creation vspace_map_pages.
  MR-based fallback handles all pipe I/O. Server handlers preserved
  for future enablement.
  Files: include/aios/root_shared.h, src/servers/pipe_server.c,
  src/lib/posix_internal.h, src/lib/posix_file.c

## Lessons Learned in v0.4.66

* **aios-cc EXTRA flags.** The script parsed -I, -D, -include into an
  EXTRA array but only passed it to the link step. Compile step got
  INCLUDES + CFLAGS but not EXTRA. This meant -include config.h never
  reached the compiler. Any external program needing custom compile
  flags (not just dash) would have hit this.

* **Quote-aware arg splitting is fundamental.** The naive split-by-space
  pattern existed in 5 separate locations in mini_shell.c. A single
  shared helper (split_args_qa) fixes all of them. The old strip_quotes
  loop ran too early, destroying argument boundary information before
  the exec path could use it.

* **seL4 frame cap = one VSpace.** A frame cap can only be mapped into
  one VSpace at a time. Mapping into a second VSpace requires
  CNode_Copy to create a fresh cap first. The original cap stays
  mapped in the first VSpace.

* **Page table leak on child VSpace mapping.** When the root task calls
  vspace_map_pages on a child VSpace, intermediate page table objects
  are allocated from the root VKA pool. sel4utils_destroy_process does
  not reclaim these extra page tables (only initial allocations). Each
  PIPE_MAP_SHM call leaked ~4-8K of untyped memory permanently. After
  3-4 pipe operations the pool was exhausted.

* **Lazy allocation matters.** Allocating the xfer frame eagerly in
  PIPE_CREATE wasted memory for pipes that never used SHM (most pipes
  in typical usage). Moving allocation to PIPE_MAP_SHM (lazy) ensures
  cost is only paid when SHM is actually requested.

## Priority 1: SHM pipes -- client-side enablement

The server-side infrastructure is complete. To enable client-side:
* Fix the page table leak. Two approaches:
  1. Pre-allocate a shared page slot during exec_server process creation
     (map a dummy page, then remap with xfer frame via seL4_ARM_Page_Map
     at a known vaddr). Page table allocated once, cleaned up normally.
  2. Track extra VSpace mappings in active_proc_t and explicitly unmap
     + free page tables during reap_forked_child.
* Re-enable pipe_request_shm() in posix_file.c once leak is fixed.
* Test: large pipe transfers should use 4096-byte chunks vs 900-byte MR.
