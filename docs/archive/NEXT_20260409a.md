## Bugs Found in v0.4.71 (not yet fixed)

### BUG 1: virtio-blk device reads return stale data

**Severity: BLOCKING for tcc -o execution**

blk_write_sector writes succeed (device acknowledges, immediate
readback correct). But subsequent blk_read_sector for the same
sector returns 0xFF (original uninitialized disk data).

Evidence:
- [pwrite] blk=8192 wrote=7f454c46 readback=7f454c46 (correct)
- [readfile] ino=390 blk0=8192 data=ffffffff (stale!)
- Happens on BOTH HVF and TCG (-accel tcg)
- QEMU cache=writethrough does not fix it
- ext2 software cache hits return correct data (write_block
  stores in cache via cache_store)
- ext2 software cache misses read from device and get 0xFF
- The 8-entry default cache is too small: tcc write phase involves
  ~1400 cache operations, evicting early blocks (including ELF header)
- Increasing cache to 256/2048 helps retain some blocks but not all
- 4096-entry cache (4MB BSS) causes memory corruption

Root cause unknown. Possibly:
- QEMU virtio-blk writeback cache not flushing to backing store
- Virtio driver avail/used ring handling after many operations
- DMA physical address mapping issue for write vs read path

Workaround that works: adding printf+fflush in pipe_server between
vfs_read and elf_newFile causes the ELF to load correctly (cache
timing effect). With this workaround, the program loads but has
Bug 2.

Files: src/boot/blk_io.c, src/ext2.c

### BUG 2: tcc-compiled programs produce no output

**Severity: BLOCKING for self-hosted development**

When a tcc-compiled program loads (via Bug 1 workaround), it
executes and returns without error, but produces NO output and
creates NO files. Both printf and open/write/close fail silently.

Evidence:
- /tmp/hello runs (no "Operation not permitted")
- No "Hello from tcc!" output
- /tmp/proof file not created
- aios-cc compiled hello_test with same source: works perfectly

Likely cause: TLS relocation produces wrong IPC buffer offset.
The seL4 IPC buffer pointer is stored in TLS. If the TP-relative
offset is wrong, all seL4_Call operations use a bad IPC buffer
address, causing silent IPC failure or VM fault.

The TLS patch in arm64-link.c computes:
  tp_off = 16 + (val - first_tls_section_addr)
GNU ld produces TP+16 for the IPC buffer (verified in aios-cc binary).
Need to verify tcc produces the same offset.

Diagnostic added: seL4_DebugPutChar('E','N','T','R','Y') at start
of __aios_entry. If visible, entry point is correct and TLS is
the issue. If not visible, entry point or __sel4runtime_load_env
fails before reaching __aios_entry.

Files: src/lib/aios_entry.c, tcc/arm64-link.c

## Lessons Learned in v0.4.71

* **FS_CAT truncates at ~952 bytes.** seL4 IPC max is ~120 MRs
  (960 bytes). FS_CAT returns file size in MR0 but only ~952 bytes
  of data. Files 953-4095 bytes were silently truncated on open.
  Fix: use FS_PREAD (multi-round-trip) for all files in openat.

* **tcc linker does not support --wrap.** The GNU ld --wrap=main
  mechanism cannot be used. Solution: custom CRT (aios_crt1.S)
  that calls __aios_entry instead of main, plus __aios_entry in
  libaios_posix that does the same cap parsing as __wrap_main.

* **tcc linker does not support ADR_PREL_PG_HI21 from GCC .o files
  as CRT input.** GCC generates ADRP+ADD for function pointer loads.
  tcc CAN resolve these (arm64-link.c handles them) but they cause
  crashes when used in CRT objects. Fix: write CRT in pure assembly
  using only R_AARCH64_CALL26 (bl) relocations.

* **musl has duplicate archive members.** syscall.o and clone.o
  appear twice in musl libc.a (assembly + C versions). ar x
  overwrites the first with the second. Fix: use ar -M with MRI
  script (addlib) to merge archives correctly.

* **munmap leak exhausts morecore.** musl malloc uses mmap for
  large allocations. munmap was a no-op, leaking virtual address
  space. tcc compilation uses ~2 mmap regions. Without reclaim,
  the 4MB morecore pool runs out during linking. Fix: LIFO
  reclaim in sys_munmap.

* **tcc needs TLS LE relocations.** seL4 runtime stores IPC buffer
  in __thread variable. GCC generates TLSLE_ADD_TPREL relocations.
  tcc arm64-link.c did not handle these. Fix: add cases for types
  549, 551, and 260.

* **ext2 block cache is critical path.** The 8-entry write-through
  cache is the ONLY reliable path for reading runtime-written data.
  Device reads (blk_read_sector) return stale data for blocks
  written at runtime. All reads of recently-written files MUST
  hit the cache.

* **PIPE_EXEC is in pipe_server, not exec_server.** The PIPE_EXEC
  handler does ELF loading inline in pipe_server.c (line 550+),
  not in exec_server.c. Debug printfs in exec_server.c never fire
  for user-initiated exec.

* **Root task printf is buffered.** The root task stdout uses musl
  stdio which is fully buffered (no terminal detection on seL4).
  printf output requires fflush(stdout) to be visible.
