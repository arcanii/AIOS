- PIPE_WRITE wakes blocked readers
- PIPE_CLOSE_WRITE wakes blocked readers with EOF
- Design is correct but caused deadlocks due to reaping issues

## Key Learnings (Hard-Won)

### 1. reap_check() NBRecv is destructive
Problem: reap_check() calls seL4_NBRecv on each childs fault_ep.
This CONSUMES the fault message. If called at the wrong time (child still
running, or child in the middle of IPC), it destroys a live process.

Rule: NEVER call reap_check() from pipe_server main loop or any server
that children actively communicate with. It was correctly removed from
fs_thread in v0.4.50 for the same reason.

### 2. PIPE_EXIT spin(100) is racy on SMP
Problem: After replying to PIPE_EXIT, the child resumes on another core
and faults. The 100-yield spin sometimes misses the fault. The child
stays active, slot never freed, waitpid defers and nobody wakes it.

Frequency: ~1 in 3 pipeline executions on 4-core SMP.

### 3. The probe-in-PIPE_WAIT approach is dangerous
Problem: Adding NBRecv probe to PIPE_WAIT (before SaveCaller defer)
catches faults from children that are still mid-execution. On SMP,
a child can send PIPE_CLOSE_WRITE, then its fault arrives on another
core before PIPE_CLOSE_WRITE is processed. The probe consumes the
fault and destroys the child before its IPC completes.

### 4. No-reply PIPE_EXIT destroys blocked children
Problem: Not replying to PIPE_EXIT leaves the child blocked in
seL4_Call. Calling sel4utils_destroy_process on a blocked TCB
sometimes corrupts CSpace state, causing cascading failures.

### 5. Printf through pipe_server corrupts pipe data
Problem: Debug printf in __wrap_main or pipe_server goes to serial_ep
IPC which competes with pipe data IPC. When stdout is redirected to
a pipe, printf output mixes with program output.

### 6. CWD not propagating through PIPE_EXEC
Problem: aios_sys_execve does not encode CWD in the buffer sent to
PIPE_EXEC. The handler defaults to "/". Fix was implemented (encode
CWD after double-null in execve buffer, extract in PIPE_EXEC) but
needs to be re-applied from clean baseline.

### 7. Redirect (> <) double-fork bug
Problem: Shell redirect code calls fork() then do_exec() which calls
fork() again internally. The inner fork creates a "(forked)" process
whose ELF path is invalid. Fix: use execv() directly in redirect child.

## Correct Architecture (For Next Session)

The fundamental problem is that pipe_server is both the IPC hub AND the
reaper. Children send PIPE_WRITE/READ/CLOSE_WRITE/EXIT to pipe_server,
but reaping requires NBRecv on fault_eps which interferes with IPC.

### Option A: Dedicated Reaper Thread
- Separate thread that blocks on child fault_eps via seL4_Recv
- When child faults, reaper thread sends a message to pipe_server
- pipe_server processes CHILD_EXITED message like any other IPC
- No NBRecv, no races, no interference with pipe I/O
- Requires: one thread per child, or a multiplexed notification

### Option B: Notification-Based Reaping
- Child fault_ep configured to also signal a notification object
- pipe_server does seL4_Poll on notification before seL4_Recv
- When notified, scan children for faulted ones
- More efficient than dedicated threads

### Option C: Keep Current Architecture, Increase Spin
- Increase PIPE_EXIT spin from 100 to 10000 (covers SMP latency)
- Add exponential backoff to avoid starving pipe_server
- Simplest change but still probabilistic

### Option D: Use seL4_ReplyRecv in PIPE_EXIT
- Instead of Reply then spin, use seL4_ReplyRecv on the childs fault_ep
- This atomically replies AND waits for the fault
- Guaranteed to catch the fault, no spinning needed
- Risk: blocks pipe_server until child faults (should be immediate)

Recommended: Option D (seL4_ReplyRecv) is the cleanest and most correct.
seL4_ReplyRecv(fault_ep, ...) replies to the PIPE_EXIT caller and
immediately blocks waiting for the next message on fault_ep, which will
be the childs VM fault. Since the child does NULL deref right after
receiving the PIPE_EXIT reply, the fault arrives within microseconds.

## Additional Findings: seL4_ReplyRecv

### seL4_ReplyRecv solves single-child reaping
Replacing the spin(100) loop with seL4_ReplyRecv(child_fault_ep)
atomically replies to PIPE_EXIT and catches the VM fault. Zero spin,
zero race. Tested 10 consecutive `ls | grep bin` with LEGACY pipe
path (EXEC_RUN) — zero freezes.

### seL4_ReplyRecv is incompatible with fork+exec pipelines
In a pipeline, multiple children (ls, grep) need pipe_server
simultaneously. seL4_ReplyRecv blocks pipe_server on child1s
fault_ep. While blocked, child2 cannot reach pipe_server for
PIPE_READ/WRITE. Result: deadlock.

Sequence that causes freeze:
1. ls (child1) sends PIPE_EXIT
2. pipe_server does seL4_ReplyRecv(child1_fault_ep) -- BLOCKS
3. grep (child2) sends PIPE_READ to pipe_ep -- BLOCKS (nobody listening)
4. ls faults, pipe_server catches it, reaps ls, returns to pipe_ep
5. grep PIPE_READ processed -- but damage already done if timing is wrong

### Intermittent fork failure: vfs_read returns -1
do_fork re-reads parent ELF from disk. vfs_read("/bin/mini_shell")
intermittently returns -1. Root cause unknown -- possibly virtio-blk
device contention or ext2 state issue. Needs investigation.

## Correct Architecture (Revised)

The fundamental constraint: pipe_server must NEVER block on anything
other than seL4_Recv(pipe_ep). Any blocking (spin, ReplyRecv, etc.)
prevents other processes from using pipe_server.

### Recommended: Notification-based fault delivery
1. Configure each child fault_ep with a notification badge
2. Bind a single notification object to pipe_servers TCB
3. seL4_Recv(pipe_ep) also receives signals from the notification
4. When a child faults, the notification fires
5. pipe_server sees the notification, scans children, reaps the faulted one
6. No blocking, no spinning, no interference with IPC

This requires seL4_Signal + seL4_TCB_BindNotification which are
standard seL4 APIs. The notification is edge-triggered so multiple
faults can be batched.

### Alternative: Bounded ntfn-augmented endpoint
seL4 allows binding a notification to an endpoint. When the endpoint
is idle (in seL4_Recv), signals from the bound notification are
delivered as regular messages with a badge. This means:
- Bind notification N to pipe_ep
- Configure child fault_eps to seL4_Signal(N) on fault
- pipe_server does seL4_Recv(pipe_ep) as normal
- When a child faults, pipe_server gets a message with fault badge
- pipe_server identifies the child, reaps it
- Zero changes to IPC flow, zero blocking
