## Completed in v0.4.67

* **SHM pipes client-side enabled.** Untyped memory leak fixed: cap copies
  from PIPE_MAP_SHM are now tracked per-pipe (xfer_copies[2]) and cleaned
  up in pipe_maybe_free via CNode_Delete + CNode_Revoke before freeing
  the xfer frame. Stale shm_vaddr bug fixed: pipe2() and close() now
  clear shm_vaddr to prevent recycled fd slots from using old mappings.
  Client-side pipe_request_shm re-enabled: lazy xfer frame allocation on
  first PIPE_MAP_SHM, up to 4096 bytes per SHM transfer vs 900 bytes MR.
  Files: include/aios/root_shared.h, src/servers/pipe_server.c,
  src/lib/posix_file.c, src/lib/posix_misc.c

* **Dash internal pipelines working.** `dash -c "echo hello | cat"`
  now completes and returns to the shell. Root cause: forked children
  that run builtins (no exec) set stdout_pipe_id locally via dup2 but
  the server never learned about the pipe assignment. When the child
  exited, handle_child_fault saw stdout_pipe_id = -1 and did not close
  the write end, so the reader (cat) never got EOF.
  Fix: PIPE_SET_PIPES IPC (label 81) sent from dup2 handler to notify
  server of stdout/stdin pipe assignments. Server updates active_proc_t
  so handle_child_fault correctly closes write/read ends on exit.
  Key insight: dash does NOT use dup2 for its output system. dash uses
  its own output.c buffering with direct fd writes. But dash DOES use
  dup2 for pipeline fd setup (redir.c copyfd), which triggers the
  PIPE_SET_PIPES notification.
  Files: include/aios/root_shared.h, src/servers/pipe_server.c,
  src/lib/posix_misc.c

* **Dash semicolons working.** `dash -c "echo a; echo b"` outputs both
  lines. mini_shell does not parse semicolons as command separators,
  so the entire string passes through to dash intact.

* **Stdin blocking (POSIX semantics).** read(0) for non-pipe stdin now
  blocks until data is available via a yield-retry loop. Previously
  aios_getchar() returned -1 immediately when the key buffer was empty,
  causing read() to return 0 (EOF). Interactive programs like dash
  exited immediately.
  Files: src/lib/posix_file.c

* **Quote-aware pipe and chain detection.** The pipe scanner (|) and
  the && / || chain handler in mini_shell now skip | inside "..." and
  single-quoted strings. Previously `dash -c "echo hello | cat"` was
  split at the | by mini_shell, producing two pipeline segments
  (`dash -c "echo hello` and `cat"`) instead of passing the full
  command to dash.
  Files: src/apps/mini_shell.c

* **Dash interactive mode working.** `dash` with no args gives a `#`
  prompt. Commands execute correctly: echo, pwd, ls, arithmetic
  expansion ($((2+3))=5). exit returns to mini_shell.
  Remaining issue: typed characters are not echoed (see Priority 1).

## Lessons Learned in v0.4.67

* **Dash must be rebuilt after libaios_posix.a changes.** Dash is
  cross-compiled separately via aios-cc. It links against
  build-04/projects/aios/libaios_posix.a. When the POSIX shim changes
  (posix_file.c, posix_misc.c, etc.), ninja rebuilds libaios_posix.a
  but does NOT rebuild dash. The stale dash binary caused hours of
  debugging -- PIPE_SET_PIPES was added to posix_misc.c but dash used
  the old version without it. Always rebuild dash after POSIX shim
  changes: `./scripts/aios-cc ... -o build-04/sbase/dash`

* **Cap copies hold untyped references.** CNode_Copy creates derived
  caps that prevent the underlying untyped from being reclaimed by
  vka_free_object. Must CNode_Delete copies and CNode_Revoke the
  original before freeing. Without this, each PIPE_MAP_SHM call
  permanently leaked the xfer frame untyped.

* **Stale fd fields on slot reuse.** aios_fd_alloc() returns inactive
  slots without zeroing them. Fields like shm_vaddr from a previous
  pipe persisted, causing PIPE_READ_SHM on freed xfer pages. Fixed
  by clearing shm_vaddr in pipe2() and close().

* **dash output.c bypasses stdout.** Dash builtins write via output.c
  which uses a file descriptor field (output.fd), not always fd 1.
  Pipeline redirect changes output.fd to the pipe write fd. This
  bypasses the POSIX shim stdout_pipe_id mechanism entirely. The
  PIPE_SET_PIPES fix works because dash still calls dup2 for pipeline
  fd wiring (close(1); dup2(pipe_fd, 1) in redir.c), even though the
  builtin later writes to a different fd.

* **tty_echo is OFF by default.** The tty_server initializes tty_echo=0
  because mini_shell implements its own character echo. Programs that
  rely on terminal echo (like dash interactive) see no typed characters.

## Priority 1: tty_server echo for dash interactive

**Problem:** Dash interactive mode works (commands execute correctly)
but typed characters are not visible. The tty_server has tty_echo=0
by default (line 83 of tty_server.c). mini_shell does its own echo
so this was never a problem.

**Root cause:** `static int tty_echo = 0;` in tty_server.c. The
echo flag is only enabled via TTY ioctl (tty_echo=1 at line 293).

**Fix approaches:**

1. **Default echo ON.** Change tty_echo=1. mini_shell would need to
   suppress double-echo (since it echoes characters itself). This
   could be done by having mini_shell send a TTY ioctl to disable
   echo before its input loop, or by removing mini_shell echo and
   relying on tty_server echo.

2. **Dash sends ioctl.** Dash normally calls tcsetattr() to configure
   terminal settings. AIOS could handle TCSETS/TCGETS ioctls to
   toggle tty_echo. This is more POSIX-correct but requires wiring
   tcsetattr through the IPC layer.

3. **Process-aware echo.** The tty_server tracks which process is
   "foreground" and enables echo for processes that need it. The
   exec_server or pipe_server signals tty_server when the foreground
   process changes.

**Recommended: Approach 1 (simplest).** Default tty_echo=1. Remove
mini_shell character echo (it uses ser_putc in the line editor).
The tty_server line discipline already handles backspace, Ctrl-U,
Ctrl-W, and all the editing that mini_shell duplicates.

Files: src/apps/tty_server.c (tty_echo default), src/apps/mini_shell.c
(remove manual echo from line editor)
