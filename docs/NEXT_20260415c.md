# AIOS Next Steps -- 2026-04-15c

Continuing from **v0.4.95** TCC self-hosting investigation.

## Session Result: VKA Reclamation Confirmed + Pipes Working

### Critical Finding: No VKA Leak

The v0.4.95 diagnosis of a VKA resource leak was **incorrect**. Detailed
instrumentation of `sel4utils_destroy_process` revealed:

- `vspace_tear_down()` walks the page table structure and calls
  `vka_utspace_free()` for every mapped frame (~3200 pages per process)
- The allocman split (buddy) allocator properly coalesces freed blocks
- `alloc_total` in `/proc/vka` remains stable across sequential commands
- The `allocated_object_list` (9 objects) only tracks page tables and
  thread infrastructure -- data frames are freed via the tear_down path

### Real Problem: Simultaneous Process Capacity

The actual issue was **capacity**, not leakage. With 12MB morecore
(3072 BSS pages per process), three simultaneous processes (required
for pipes: shell + cmd1 + cmd2) exceed the 8000-page VKA pool:

    3 x 3222 pages = 9666 > 8000 pool

### Fix: Morecore 12MB -> 6MB

Reducing morecore to 6MB (1536 pages/process) allows 3-4 simultaneous
processes within the 8000-page pool:

    3 x ~1700 pages = ~5100 < 8000 (single pipe)
    4 x ~1700 pages = ~6800 < 8000 (double pipe)

## Next Session Priorities

### Priority 1: Demand-Paged mmap (MAP_ANONYMOUS)

The real solution to per-process memory. Current state:
- 6MB BSS eagerly mapped = 1536 pages per process
- Prevents >4 simultaneous processes
- TCC linker needs 15-20MB but 6MB is the max

With demand paging:
- Process starts with 256KB-1MB mapped
- mmap(MAP_ANONYMOUS) allocates VKA frames on demand
- Most programs (ls, cat, echo) never exceed 1MB
- TCC linker can grow to 15-20MB as needed
- munmap and exit cleanup free frames back to VKA

Implementation:
1. Reduce default morecore to 1-2MB
2. Add mmap/munmap syscall handlers (new posix_mmap.c)
3. Root task handles MAP_ANONYMOUS via VKA frame allocation
4. Track mapped regions per-process for cleanup on exit
5. Test: TCC link step succeeds natively

### Priority 2: Pipe SHM Frame Cleanup

Minor leak: pipe SHM frames (1 page each) accumulate across pipe
operations. The `pipe_t.shm_frame` is allocated in PIPE_CREATE but
not always freed when the pipe is destroyed. Visible in /proc/vka
as growing `pipe: fr=N` count.

### Priority 3: TCC Native Link Step

Blocked by demand paging (Priority 1) and:
- libc.a archive format (duplicate members)
- SHM file read path for large archive I/O
- Proper libc.a rebuild with new morecore

## Key Learnings

### sel4utils_destroy_process IS Complete

The seL4 utils library handles full resource cleanup:
1. `vka_cnode_revoke` -- revokes child CNode (all child caps)
2. `sel4utils_clean_up_thread` -- frees TCB, IPC buffer, stack
3. `vspace_tear_down` -- walks page table, frees ALL data frames via
   `vka_utspace_free` (buddy allocator coalescence)
4. `clear_objects` -- frees page tables and thread objects from
   `allocated_object_list` (only ~9 items)
5. `vka_free_object` on fault endpoint and page directory

### allocated_object_list is NOT the Main Cleanup Path

The `allocated_object_list` (freed by `clear_objects`) only contains
~9 objects per process: page tables + thread infrastructure. The
~3200 data frames are tracked in the vspace's internal page table
structure and freed by `vspace_tear_down` -> `free_pages_at_level`
-> `sel4utils_unmap_pages` -> `vka_utspace_free`.

### TCC Cross-Compilation Must Use Two-Step Build

TCC source expects glibc headers at compile time. Cross-compiling
with aios-cc (which provides musl headers) produces a broken binary.
Correct approach:
1. `aarch64-linux-gnu-gcc -c -DONE_SOURCE=0` (host headers)
2. `aarch64-linux-gnu-ld -r` (relocatable link into tcc_all.o)
3. `aios-cc tcc_all.o -o tcc` (final link with musl/AIOS runtime)
