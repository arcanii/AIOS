## Session Result: RPi4 Serial Boot + Dynamic Hardware Banner

### Problem
RPi4 boot broken since v0.4.93. Multiple root causes identified and fixed.

### Root Causes Found

1. **ARM64 Image header required**: RPi firmware rejects raw binaries
   without "ARMd" magic at offset 0x30. Previous sessions used the
   diagnostic stub which had compatible entry code.

2. **Firmware relocates to 0x80000**: Regardless of kernel_address or
   text_offset, firmware always relocates ARM64 images to 0x80000.
   seL4 elfloader is linked at ~0x370c000 -- crashes at 0x80000.

3. **SMP not ready**: KernelMaxNumNodes=4 causes silent boot failure
   (secondary cores not parked).

4. **GENET driver crashes**: 731-line driver from rpi4 branch crashes
   during 64KB MMIO mapping on real hardware.

5. **Display driver crashes**: VKA allocator assertion when mapping
   VC mailbox registers. Phase A fails (no diagnostic stub fb_info).

6. **RAM overflow**: int total_mem overflows at >2GB (RPi4 has 3.9GB).

7. **Stale ext2 binaries**: Disk had QEMU-compiled getty without
   platform detection.

8. **Hardcoded banner strings**: Getty used compile-time ifdefs
   instead of runtime hardware detection.

### Additional Findings

- mtools FAT32 (mformat/mcopy) breaks RPi firmware FAT32 parser.
  Must use macOS diskutil for boot partition.
- Flash procedure: mksdcard.py dd (full image) -> diskutil eraseVolume
  P1 (reformat FAT32) -> cp boot files. ext2 on P2 survives P1 reformat.
- No dtoverlay=disable-bt: GPIO 14/15 must be mini UART for root task.
- core_freq=250 + core_freq_min=250: stabilizes mini UART baud rate.
