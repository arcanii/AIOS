# AIOS Next Steps -- 2026-04-15d

Continuing from **v0.4.97** RPi4 boot investigation.

## Session Result: RPi4 Serial Boot Restored

### Problem
RPi4 boot broken since v0.4.93. Three root causes identified and fixed:

1. **ARM64 Image header required**: RPi firmware now rejects raw binaries
   without "ARMd" magic at offset 0x30. Previous sessions used the
   diagnostic stub which had compatible entry code.

2. **Firmware relocates to 0x80000**: Regardless of kernel_address or
   text_offset config.txt settings, the firmware always relocates ARM64
   images to 0x80000. seL4 elfloader is linked at ~0x370c000 and crashes
   at 0x80000 because kernel load overwrites the elfloader.

3. **Unstable drivers**: GENET Ethernet (731-line driver from rpi4 branch)
   crashes during MMIO mapping. VideoCore display driver asserts in VKA
   allocator when mapping mailbox registers.

### Fixes Applied

| Fix | File | Change |
| --- | --- | --- |
| ARM64 Image header | hw/rpi4/relocator.S | 4KB stub with "ARMd" magic |
| Relocator stub | hw/rpi4/relocator.S | Copies payload to link address |
| SMP disabled | settings-rpi4.cmake | KernelMaxNumNodes 4 -> 1 |
| GENET disabled | src/plat/rpi4/net_genet.c | plat_net_init returns -1 |
| Display disabled | src/plat/rpi4/display_vc.c | plat_display_init returns -1 |

### Additional Findings

- **FAT32 format matters**: RPi firmware cannot read mtools-generated FAT32.
  Must use macOS diskutil (Method 2) for boot partition.
- **No disable-bt overlay**: GPIO 14/15 must be mini UART (ALT5) for root
  task serial I/O. The disable-bt overlay switches to PL011 which the root
  task does not use.
- **Same firmware files**: start4.elf from v0.4.93 works. The firmware did
  not change; the missing diagnostic stub caused the header check to fail.
