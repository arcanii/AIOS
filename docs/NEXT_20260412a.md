## Completed in v0.4.83

### SSH Phase 3: Encrypted Transport

- `ssh_encrypt.c`: AES-256-CTR stream cipher + HMAC-SHA-256 integrity
- mbedTLS `mbedtls_aes_crypt_ctr()` for both encrypt and decrypt (same CTR op)
- HMAC-SHA-256 implemented directly from SHA-256 primitives (RFC 2104)
- Constant-time MAC comparison (prevents timing side-channel)
- Cipher state persists across packets (CTR stream is continuous)
- Padding block alignment changed from 8 to 16 after NEWKEYS (AES block size)
- `ssh_transport.c`: encrypted read/write paths dispatched via `s->encrypted` flag
- `ssh_session.h`: IV 12->16 bytes (AES-CTR needs full block), added MAC keys (E/F)
- `ssh_crypto.c`: key derivation stores 16-byte IVs + 32-byte MAC keys (letters A-F)
- `ssh_kex.c`: strict KEX post-NEWKEYS sequence number reset, removed premature `encrypted=1`
- Verified: OpenSSH 10.2 SERVICE_REQUEST decrypted, SERVICE_ACCEPT accepted by client

### SSH Phase 4: User Authentication

- `ssh_auth.c`: SSH USERAUTH protocol (RFC 4252)
- Handles method="none" probe -> responds USERAUTH_FAILURE listing "password"
- Handles method="password" -> AUTH_LOGIN IPC to auth_server (same protocol as getty)
- String packing into seL4 message registers (8 bytes per MR, little-endian)
- PIPE_SET_IDENTITY IPC to pipe_server after successful login (uid/gid propagation)
- Password scrubbed from stack after use
- Max 3 auth attempts before disconnect
- Uses `aios_get_auth_ep()` / `aios_get_pipe_ep()` from aios_posix.h
- Verified: OpenSSH password auth, AUTH_LOGIN OK uid=0 gid=0 token=0x1003

### TCP bug fix: data loss on partial blocked reads

- **Root cause**: when a TCP segment arrived and the blocked reader wanted fewer bytes than available (e.g., 4-byte SSH header from a 160-byte segment), only the requested bytes were delivered. The remaining bytes were silently discarded.
- **Fix**: after waking the blocked reader, buffer excess data in the socket circular buffer
- **Location**: `net_server.c` TCP ESTABLISHED data receive path
- **Impact**: affected any protocol doing small sequential reads on TCP sockets

### Strict KEX sequence number fix

- Added post-NEWKEYS sequence number reset (`seq_recv=0, seq_send=0`)
- Moved `encrypted=1` flag from ssh_kex.c to ssh_encrypt_init() (correct ordering)
- Ensures first encrypted packet uses seq=0 for MAC computation

## Completed in v0.4.84

### SSH Phase 5: Session Channel + Shell (RFC 4254)

- `ssh_channel.c` (new, ~450 lines): full channel lifecycle
  - SSH_MSG_CHANNEL_OPEN "session" + CHANNEL_OPEN_CONFIRMATION
  - SSH_MSG_CHANNEL_REQUEST "pty-req" (terminal size stored), "env" (accepted), "shell" (triggers spawn)
  - Shell spawn: pipe2 x2, fork, dup2, execv `/bin/dash -i`
  - Bidirectional data relay: polls pipe (O_NONBLOCK) + socket (non-blocking inbuf accumulation)
  - Server-side line discipline: echo, backspace, Ctrl-C (SIGINT), Ctrl-D (EOF), CR->LF conversion
  - LF->CRLF conversion on shell output for SSH terminal display
  - CHANNEL_WINDOW_ADJUST flow control
  - CHANNEL_DATA, CHANNEL_EOF, CHANNEL_CLOSE handling
- `ssh_transport.c`: buffer-based `ssh_read_packet_nb()` -- accumulates raw bytes in session inbuf, decrypts header once to learn length, processes only when complete packet available. Never blocks.
- `ssh_session.h`: channel window/pty state, non-blocking reassembly fields (nb_dec_hdr, nb_pkt_len, nb_hdr_done)
- `sshd_main.c`: integrated `ssh_do_channel()` after auth (replaced disconnect placeholder)
- Verified: OpenSSH 10.2 interactive shell -- ls, uname -a, cat, whoami all working

### O_NONBLOCK for TCP sockets

- `net_server.c`: reads MR2 as nonblock flag in NET_RECVFROM handler; returns -EAGAIN (-11) instead of SaveCaller blocking when flag set and no data available. Both TCP and UDP paths.
- `posix_file.c`: passes `is_nonblock` flag in MR2 for socket reads (3 MRs instead of 2)
- Enables non-blocking TCP socket reads needed for SSH relay polling

### PIPE_READ_SHM false-EOF fix

- **Root cause**: PIPE_READ_SHM returns 0 for both real EOF (write_closed) and `!xfer_valid` failure. Client cannot distinguish the two cases.
- **Fix**: `posix_file.c` -- when SHM read returns 0 with O_NONBLOCK set, fall through to MR-based PIPE_READ path which correctly returns -EAGAIN vs 0.

### PIPE_EXEC pipe reference counting fix

- **Root cause**: PIPE_EXEC never incremented write_refs/read_refs when assigning pipes to a new process. After fork+exec, child and parent shared write_refs=1. First child exit set write_closed=1, permanently closing the pipe for all processes.
- **Fix**: `pipe_server.c` -- PIPE_EXEC increments write_refs/read_refs when setting stdout_pipe_id/stdin_pipe_id on the new process.
- **Impact**: external commands (ls, uname, cat) now produce output through SSH pipes correctly.

### PIPE_EXEC TCB destruction fault suppression

- **Root cause**: `sel4utils_destroy_process()` during PIPE_EXEC generates a fault on pipe_ep. This fault arrives after active_procs entry is overwritten with new process data, causing `handle_child_fault` to close the NEW process's pipes.
- **Fix**: `ignore_next_fault` flag in active_proc_t. Set before destroy, checked and cleared in handle_child_fault.
- Added field to `root_shared.h` active_proc_t

### Fork+exec pipe close fix

- Child process must NOT call close() on pipe write ends before exec. In AIOS, close() on a pipe write end sends PIPE_CLOSE_WRITE to pipe_server, decrementing write_refs. After fork, child closing unused ends prematurely sets write_refs=0. Since exec replaces the process entirely via PIPE_EXEC, the old fds are irrelevant.
- Parent must NOT close out_pipe[1] during relay (same reason). Closed only at relay cleanup.

## Known Issues

### Ctrl-C behavior (SSH + local)

- Ctrl-C sends SIGINT via kill() + writes 0x03 to shell stdin
- Signal delivery to pipe-redirected processes depends on AIOS signal model
- May not interrupt long-running child commands (uname, cat) promptly
- Local Ctrl-C regression still present (possibly UART-related)

### Shell exit detection

- Pipe EOF detection uses `ever_got_output` guard: ignores spurious pipe read=0 before shell produces any output (caused by PIPE_READ_SHM xfer_valid race)
- After shell produces output, pipe read=0 is treated as real EOF -> sends CHANNEL_EOF + CHANNEL_CLOSE
- Relies on pipe_server correctly cleaning up write_refs on shell exit

### net_server.c debug prints

- Three debug printf statements remain (NTFN-WAKE, reader blocked, wake blocked reader)
- Remove once TCP stability is confirmed

### TCP reliability

- No retransmission (occasional decryption errors from lost TCP segments via QEMU SLIRP)
- No window advertisement
- Reconnect if session drops with `enc-read: bad pkt_len`
