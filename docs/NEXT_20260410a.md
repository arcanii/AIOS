# AIOS Next Steps -- 2026-04-10a

## Starting from v0.4.73

## Completed in v0.4.73

* **Second virtio-blk drive for logging.** AIOS now discovers multiple
  block devices at boot. Each device is probed by reading its ext2
  superblock and checking the volume label (s_volume_name at offset
  0x78). Drives labeled "aios-log" are mounted at /log as a dedicated
  log filesystem. Drive order in the QEMU command does not matter.
  Files: src/boot/boot_fs_init.c (rewritten), src/boot/boot_log_init.c (new)

* **ext2 per-device block cache.** The 2048-entry block cache is now
  keyed by (dev_id, block_number). Each ext2_ctx_t carries a dev_id
  field (0=system, 1=log). Cache lookups and stores check dev_id to
  prevent cross-drive collisions. ext2_init takes a dev_id parameter.
  Files: include/aios/ext2.h, src/ext2.c

* **ext2 inode_size from superblock.** ext2_read_inode and write_inode
  now use s_inode_size from the superblock (rev 1+) instead of
  hardcoded 128. Handles both 128-byte (AIOS builder) and 256-byte
  (mke2fs default) inodes correctly.
  Files: include/aios/ext2.h, src/ext2.c

* **File-based logging (/log/aios.log).** aios_log_write now outputs
  to three destinations: ring buffer (16KB), serial, and a file on
  the log drive. The file is created/opened by aios_log_file_init()
  called from fs_thread_fn at server startup. Boot entries are
  captured by flushing the ring buffer to the file after init.
  Direct ext2_pwrite_file calls bypass VFS/IPC to avoid deadlocks.
  Files: include/aios/aios_log.h, src/aios_log.c, src/servers/fs_server.c

* **ext2_alloc_block group_start fix.** For block groups >= 1, the
  group start was computed as g * blocks_per_group, missing the
  first_data_block offset. For 1K blocks (first_data_block=1), group 1
  returned block 8192 instead of 8193 -- colliding with the last block
  of group 0. Fix uses Linux kernel formula: g * bpg + first_data_block.
  File: src/ext2.c

* **ext2_create_file 0755 permissions.** Changed hardcoded inode.i_mode
  from 0x81A4 (-rw-r--r--) to 0x81ED (-rwxr-xr-x). Runtime-created
  files (tcc output, shell scripts) are now executable.
  File: src/ext2.c

* **virtio-blk post-completion barrier.** Added dmb sy after the
  polling loop in blk_read_sector and blk_write_sector, before reading
  DMA data. On Apple Silicon HVF (weak memory model) and TCG (I/O
  thread vs vCPU thread), device data writes may not be globally
  visible when used->idx changes. This partially mitigates the stale
  read bug (BUG 1 from v0.4.71).
  File: src/boot/blk_io.c

* **ext2_pread_file and ext2_pwrite_file declared in ext2.h.** These
  functions were defined in ext2.c but not exposed in the header.
  File: include/aios/ext2.h

* **Log drive I/O functions.** blk_read_sector_log and
  blk_write_sector_log use separate DMA buffers for the second
  virtio-blk device. Includes post-completion dmb sy barrier.
  Files: src/boot/blk_io.c, include/aios/root_shared.h, src/aios_root.c

## Bugs Still Open

### URGENT: Ctrl-C cannot interrupt blocking read loops

**Severity: HIGH -- user-facing hang, requires QEMU kill to recover**

Programs in infinite read loops (tail -f, blocking cat on pipes, any
program waiting for input that will never arrive) cannot be interrupted
by Ctrl-C. The terminal appears frozen. Only QEMU termination recovers.

Root cause: Ctrl-C in the main loop calls process_kill() which destroys
the child process via sel4utils_destroy_process. This works for running
programs but NOT for programs blocked on IPC (seL4_Recv on the serial
or fs endpoint). The blocked IPC is in the tty_server or fs_server
address space, not the child process -- destroying the child VSpace
does not unblock the IPC.

For tail -f specifically: sbase tail with -f enters a loop calling
read() which sends FS_PREAD IPC. The IPC completes (returns 0 bytes
at EOF), but tail loops calling nanosleep + read forever. The read
IPC is NOT blocking -- the issue is that tail never checks for signals
between iterations. The POSIX shim needs to check for pending signals
in the nanosleep or read path and return EINTR.

Investigation paths:
* Add PIPE_SIG_FETCH check in sys_nanosleep (posix_time.c)
* Add PIPE_SIG_FETCH check in sys_read for non-pipe fds (posix_file.c)
* Consider a signal-pending flag in the CWD metadata that programs
  can check without IPC overhead
* For truly blocked IPC (seL4_Recv): need async signal via fault EP
  injection or a timeout-based poll loop

Files: src/lib/posix_time.c, src/lib/posix_file.c,
       src/servers/pipe_server.c, src/lib/posix_proc.c

### BUG 1: virtio-blk stale reads (partially mitigated)

The dmb sy barrier added in v0.4.73 addresses the memory ordering
issue between the device completion signal (used->idx) and the DMA
data buffer. The printf+fflush workaround from v0.4.72 is no longer
needed. However, the stale read issue may still occur under heavy
cache eviction pressure. Needs stress testing with repeated tcc
compilation cycles.

Status: partially mitigated, monitor during tcc self-hosting tests.
Files: src/boot/blk_io.c

## Lessons Learned in v0.4.73

* **QEMU virt assigns virtio-mmio devices in reverse command-line
  order.** The last -device on the command line gets the lowest MMIO
  slot number. Boot code that scans from slot 0 finds the LAST device
  first. Fix: identify drives by content (volume label), not slot order.

* **mke2fs default inode size is 256 bytes on modern e2fsprogs.**
  AIOS ext2.c hardcoded 128-byte inodes. Drives formatted by mke2fs
  had misaligned inode reads, returning garbage for mode/size fields.
  Fix: read s_inode_size from the superblock at offset 88 (rev 1+).

* **ext2_alloc_block group_start off-by-one for non-zero groups.**
  The formula (g==0)?1:g*bpg missed first_data_block for groups >= 1.
  Linux kernel reference: ext2_group_first_block_no() always adds
  first_data_block. This caused allocated blocks to collide with the
  last block of the previous group.

* **Calling ext2_mkdir from the boot chain overflows the root task
  stack.** ext2_alloc_block uses 3072 bytes of local buffers. The boot
  call chain (main -> boot_fs_init -> boot_log_drive_init) adds ~2KB.
  Total ~5KB exceeds the root task stack. Fix: do not create mount
  point directories at boot; VFS prefix matching handles dispatch
  without a real directory on the parent filesystem. Use disk/rootfs
  overlay to create the mount point directory in the disk image.

* **Forward declarations matter in C.** Calling a static function
  before its definition without a forward declaration silently fails
  with -Wimplicit-function-declaration treated as error. The build
  succeeds with the old object file, masking the missing code.

* **Per-device cache namespace is essential for multi-drive ext2.**
  Without dev_id in cache entries, block N on the system disk and
  block N on the log disk would alias in the cache, causing data
  corruption.

## Priority 0: Fix Ctrl-C signal delivery

See URGENT bug above. Minimum viable fix:
1. In sys_nanosleep (posix_time.c): check PIPE_SIG_FETCH before sleep
2. If signal pending, return -EINTR
3. This unblocks tail -f which loops on nanosleep+read

For robust signal delivery to blocked IPC, longer-term:
4. Add timeout to seL4_Recv in child process read paths
5. Or use notification-based wakeup (seL4_Signal to child endpoint)

## Priority 1: Stress test stale read fix

Run repeated tcc compilation cycles to trigger cache eviction.
Verify that the dmb sy barrier prevents stale reads.
