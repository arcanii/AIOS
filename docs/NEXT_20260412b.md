## Completed in v0.4.85

### Pipe subsystem fixes

- **Critical: `ignore_next_fault` not cleared in PIPE_EXEC** (v0.4.84 bug)
  - Root cause: PIPE_EXEC sets `ignore_next_fault=1` before `sel4utils_destroy_process()` to suppress the destruction fault. But the old process TCB is deleted while blocked, so the fault is never delivered and the flag never cleared. The NEW process exit fault is silently discarded.
  - Impact: every fork+exec child process was never reaped. PIPE_WAIT deadlock on all pipe commands. SSH tests did not expose this because ssh_channel relay does not use PIPE_WAIT.
  - Fix: `ap->ignore_next_fault = 0;` after PIPE_EXEC process setup (pipe_server.c)

- **PIPE_READ_SHM returns -2 for `!xfer_valid`** (was 0, ambiguous with EOF)
  - Server: split active/xfer_valid checks, return -2 for xfer not ready
  - Client (posix_file.c): on -2, fall through to MR-based PIPE_READ path
  - Removes the v0.4.84 false-EOF workaround (nonblock + got==0 fallthrough)

- **PIPE_CLOSE_READ conditional `read_closed`**
  - Only set `read_closed=1` when `read_refs` drops to 0 (was unconditional)
  - Same fix in `handle_child_fault` stdin pipe path
  - Prevents premature EPIPE in multi-reader scenarios

- **PIPE_DUP_REFS (label 82)** handler added to pipe_server
  - Accepts pipe_id + flags (bit0=read, bit1=write), increments refs
  - For future explicit per-process pipe ref tracking after fork
  - Not auto-called in clone (PIPE_EXEC already handles fork+exec refs)

### Ctrl-C signal delivery fixes

- **Two-stage Ctrl-C uses root-side `fg_sigint_sent` flag**
  - Old: checked `sig_pending` bit for second Ctrl-C detection. Failed when process cleared sig_pending via PIPE_SIG_FETCH between presses.
  - New: root task tracks `fg_sigint_sent` independently. First Ctrl-C sets flag + sig_pending. Second Ctrl-C checks flag and force-kills.
  - Flag reset when foreground process exits (handle_child_fault)

- **Force-kill via PIPE_SIGNAL(SIGKILL) instead of direct `sel4utils_destroy_process`**
  - Old: root task called `sel4utils_destroy_process` directly, corrupting capabilities (null cap crash)
  - New: sends `PIPE_SIGNAL(fg_pid, 9)` IPC to pipe_server. pipe_server SIGKILL handler calls `handle_child_fault` which properly reaps, wakes PIPE_WAIT, and cleans up.

- **PIPE_WAIT interrupted by pending signal**
  - PIPE_SIGNAL handler now checks if target process is blocked in PIPE_WAIT
  - If blocked: unblocks waiter with -EINTR so it can dispatch the signal
  - Enables signal delivery to processes stuck in waitpid()

- **sshd ignores SIGINT**
  - `signal(SIGINT, SIG_IGN)` + `#include <signal.h>` in sshd_main.c
  - Prevents local Ctrl-C from killing sshd on first press
  - Second Ctrl-C force-kills via PIPE_SIGNAL(SIGKILL)

### SSH fixes

- `ssh_session.h`: added missing `has_peek`/`peek_byte` fields in ssh_session_t
  - Required for encrypted packet read lookahead in ssh_transport.c

### Network fixes

- `net_server.c`: removed 7 debug printf statements (SYN-ACK, ESTABLISHED, wake reader, FIN, ready, NTFN-WAKE, blocked)
- `posix_net.c`: `recvfrom()` now passes `is_nonblock` flag in MR2 (was always 0). Server-side O_NONBLOCK support was already present but client never sent the flag.

## Known Issues

### Socket cleanup after force-kill

- When sshd is force-killed (SIGKILL), its listen socket on port 2222 is not freed in net_server
- Restarting sshd after force-kill may crash (stale socket state, null cap error)
- Fix: associate sockets with PID in net_server, cleanup in handle_child_fault

### TTY echo timing with pipes

- Pipe commands occasionally show garbled output ("cathello" instead of newline + "hello")
- Caused by tty_server character echo overlapping with command output
- Cosmetic issue, pipe data is correct

### SSH Ctrl-C to child commands

- SSH relay sends kill(shell_pid, SIGINT) on Ctrl-C
- Shell (dash) catches SIGINT but does not forward to running child command
- Needs process group support or direct child PID tracking in SSH relay

### Ctrl-C double prompt

- After Ctrl-C, tty_server pushes newline to line_queue which dash reads as empty command
- Causes an extra `# ` prompt after Ctrl-C. Cosmetic.
