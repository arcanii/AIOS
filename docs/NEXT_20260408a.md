## Completed in v0.4.65

* **Shared-memory pipe buffers.** Each pipe now allocates a 4K frame via
  vka_alloc_frame, mapped into the root VSpace. The ring buffer pointer
  (shm_buf) replaces the static buf[] array. Data still flows through IPC
  message registers, but storage is a proper mapped frame. Cleanup unmaps
  and frees on ref drop. Fallback to static buf if alloc fails.
  Files: include/aios/root_shared.h, src/servers/pipe_server.c

* **VKA allocator audit.** Per-subsystem allocation counters (boot, fork,
  exec, thread, pipe) track frames, endpoints, TCBs, cslots, and untypeds.
  Dump via shell `debug` command. Instrumented: boot_services.c (7 endpoints),
  boot_fs_init.c (DMA untyped), aios_root.c (fault_ep, serial_ep), fork.c
  (frame + cslot allocs), thread_server.c (TCB, fault_ep, IPC frame, stack),
  exec_server.c (cslots), pipe_server.c (shm frame).
  Files: NEW include/aios/vka_audit.h, NEW src/lib/vka_audit.c,
  plus instrumentation in 7 existing files

* **O_APPEND support (>> redirect).** Shell detects >> and reads existing
  file content via FS_CAT IPC before writing combined old+new data via
  FS_WRITE_FILE. Also added is_append flag to aios_fd_t and openat sets
  pos=size when O_APPEND is set (for future VFS-level append).
  Files: src/apps/mini_shell.c, src/lib/posix_internal.h, src/lib/posix_file.c

* **Pipe cleanup fix.** PIPE_CLOSE now uses ref-count path (pipe_maybe_free)
  instead of direct dealloc, preventing corruption of VSpace mappings on
  subsequent pipe allocations.
  Files: src/servers/pipe_server.c

* **Build: vka_audit.c added to aios_root target** (not aios_posix -- it
  uses VKA/vspace types only available in the root task link).
  Files: projects/aios/CMakeLists.txt

## VKA Audit Results

Measured allocations after boot + login + several commands:

```
subsys   frames   eps tcbs cslots untypd   tot_pg
boot          0     7    0      0      1        4
fork          0     0    0   1270      0        0
exec          0     0    0      4      0        0
pipe          2     0    0      0      0        2
TOTAL                                           6
pool = 4000 pages, remaining ~ 3994 pages
```

Key findings:
* Pool is ~100x oversized. 4000 pages = 16MB, only 6 pages used.
* Fork cost is cslot-dominated (capability duplication), not frame-dominated.
  sel4utils_configure_process_custom allocates frames internally (not via
  our vka_alloc_frame calls), so those are not captured by the audit.
* Boot uses 7 endpoints (fs, exec, thread, auth, pipe, fault, serial)
  plus 4 pages for DMA ring buffer.
* Each pipe uses 1 page for shm buffer.
* Thread creation (TCB + fault_ep + IPC frame + 4 stack frames) would add
  ~6+ pages per thread but no threads were created in this test.

## Lessons Learned in v0.4.65

* **Disk image rebuild required for app changes.** mini_shell and all apps
  in src/apps/ are loaded from the ext2 disk image, not from the kernel
  binary. After editing mini_shell.c, must run mkdisk.py to update the
  disk. Failing to do this means QEMU runs stale binaries. This caused
  significant debugging time in this session.

* **Full rebuild (rm -rf build-04) deletes sbase.** The sbase cross-compiled
  binaries live in build-04/sbase/. A full clean rebuild requires re-running
  build_sbase.py before mkdisk.py. Missing sbase means /bin/echo, /bin/cat
  etc. are absent and the shell reports "command not found" for everything.

* **VKA audit placement matters.** vka_audit.c uses only printf and stdint
  types, but the instrumentation calls live in root task code that links
  against VKA/vspace. The audit .c must be in the aios_root CMake target,
  not in aios_posix (which is a client-side static library).

* **FS_WRITE_FILE replaces file content.** ext2_vfs_create does unlink+create,
  so O_APPEND cannot be implemented by simply setting file position. Must
  read existing content and write old+new concatenated. Future: add a
  dedicated FS_APPEND IPC op.

* **IPC label values.** FS_CAT = 11 (not 2). Always reference the label
  definitions in root_shared.h or posix_internal.h rather than hardcoding.

1. mini_shell quote parsing: single quotes may be stripped or split

## Priority 3: VFS-level append

Current O_APPEND does read-modify-write at the shell level. This is
inefficient and fragile. Add FS_APPEND IPC op to fs_server that reads
existing content server-side and appends new data in a single operation.

This would allow posix_file.c write() to use FS_APPEND directly when
is_append is set, without the shell needing special handling.

IPC design: FS_APPEND = 19 (next available label after FS_RENAME=18)
* MR0 = path_len
* MR1+ = path bytes
* Then data_len + data bytes (same format as FS_WRITE_FILE)
* Server reads existing file, appends new data, writes combined

Files: src/servers/fs_server.c, src/lib/posix_file.c,
include/aios/root_shared.h (new label)

## Priority 4: Shared-memory pipes phase 2

Current shm pipes map the buffer into root VSpace only. Data still
copies through IPC message registers. Phase 2:
* Map the shm frame into writer and reader process VSpaces
* Writer writes directly to shared page (no IPC for data transfer)
* IPC used only for signaling (length written, EOF)
* Requires passing mapped vaddr back to client during pipe2()
* Client-side posix_file.c pipe read/write uses memcpy to/from shared page

This would eliminate the 900-byte-per-IPC bottleneck for pipe data transfer.

Files: src/servers/pipe_server.c, src/lib/posix_file.c,
src/lib/posix_internal.h (pipe fd needs shm_vaddr field)
