## Key Bugs Found and Fixed

### 1. pipe2() fd allocation (THE root cause)

Problem: aios_fd_alloc() returned free slot but did not mark it active.
pipe2() called it twice, both returned the SAME index. Read fd == write fd.
ALL data written was immediately consumed by the same fd.
This was the root cause of every pipe failure across 20+ attempts.

Fix: Mark aios_fds[ri].active = 1 immediately after first alloc,
before calling aios_fd_alloc() for the second time.

### 2. close() sending PIPE_CLOSE destroyed pipe buffer

Problem: close(write_fd) sent PIPE_CLOSE to pipe_server, which
zeroed the entire pipe buffer. Reader never saw any data.

Fix: close() never sends server IPC. Only aios_exit_cb sends
PIPE_CLOSE_WRITE (for EOF). Shell sends PIPE_CLOSE after pipeline.

### 3. close(write_fd) sent PIPE_CLOSE_WRITE prematurely

Problem: In fork+dup2+exec, child does dup2(write_fd, 1) then
close(write_fd). If close sent PIPE_CLOSE_WRITE, EOF fired
before exec even loaded the program.

Fix: close() does not send any pipe IPC at all. The exit path
handles write-end EOF deterministically.

### 4. SMP fork startup race

Problem: On 4-core SMP, parent forks child A and child B rapidly.
Both children call PIPE_EXEC simultaneously. pipe_server processes
them concurrently, causing state corruption (shared elf_buf, etc).

Fix: 20x seL4_Yield() between forks in run_pipeline. Proper fix
(next session): completion handshake in PIPE_FORK/PIPE_EXEC.

### 5. MAX_ACTIVE_PROCS exhaustion

Problem: 8 slots minus 7 servers = 1 slot for children.
Pipelines need 2-3 slots simultaneously.

Fix: MAX_ACTIVE_PROCS increased from 8 to 16.

## Architecture Insight

The correct Unix pipe pattern on seL4 is:
1. Parent calls pipe() -- creates server-side buffer, returns two fds
2. Parent forks children (with yield between forks on SMP)
3. Each child: dup2(pipe_fd, 0 or 1), close all pipe fds, execv()
4. Parent: close all pipe fds, waitpid all children
5. Parent: PIPE_CLOSE to destroy server-side buffer

Key insight: close() in child must NOT signal the pipe server.
Only the exit callback (process death) signals PIPE_CLOSE_WRITE.
This matches Unix semantics where close(fd) just decrements a
reference count, and the pipe stays alive until all writers exit.
