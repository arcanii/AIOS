# AIOS Next Steps -- 2026-04-11a

## Completed in v0.4.75

### Display: ramfb framebuffer (1024x768)

- QEMU ramfb device via fw_cfg DMA interface
- fw_cfg MMIO at 0x09020000: file enumeration, DMA write
- Framebuffer: 1024 contiguous pages (4MB), XRGB8888 format
- boot_display_init.c: fw_cfg probe, ramfb config, framebuffer alloc
- QEMU command: `-device ramfb` (replaces `-nographic`)

### Display: 8x8 bitmap font + boot splash text

- Embedded 8x8 bitmap font (ASCII 32-126, 95 glyphs, 760 bytes)
- Scalable text rendering: gpu_draw_char(x, y, ch, color, scale)
- Boot splash: "AIOS v0.4.75" at 3x scale + system info at 1x
- Optional splash image from /images/splash.raw on disk

### Display: PNG-to-raw converter

- scripts/png_to_raw.py: converts PNG/JPEG to AIOS raw format
- Raw format: 16-byte header (width, height, format, reserved) + XRGB pixels
- Requires Pillow: `pip3 install Pillow`
- Default output: disk/rootfs/images/splash.raw

### Display server IPC (5 commands)

- display_server.c: IPC server thread in root task
- disp_ep endpoint propagated to all child processes
- Argv layout changed: 8 cap args (was 7), __wrap_main shift 8 (was 7)
- IPC labels 110-114:
  - DISP_FB_INFO (110): returns width, height, active
  - DISP_SHOW_FILE (111): server reads raw file from VFS, blits to FB
  - DISP_TEXT (112): render text at position with font
  - DISP_FILL_RECT (113): fill rectangle with color
  - DISP_CLEAR (114): clear screen to color

### fbshow user program

- `fbshow /images/splash.raw` -- display raw image (centered)
- `fbshow --info` -- print framebuffer dimensions
- `fbshow --clear` -- clear to black
- `fbshow --clear RRGGBB` -- clear to hex color

### UART IRQ notification

- PL011 RX interrupt (IRQ 33) bound to seL4 notification
- Main loop uses seL4_Wait instead of seL4_Yield (zero CPU when idle)
- seL4_IRQHandler_SetNotification + seL4_IRQHandler_Ack cycle
- Fallback to polling if IRQ setup fails
- NOTE: do NOT use seL4_TCB_BindNotification -- it interferes with
  seL4_Call by returning notification badge instead of expected reply

### Pipe SHM server-side optimization

- PIPE_WRITE_SHM: simplified loop (compute avail first, single count update)
- Semantically equivalent, reduces per-byte overhead

## Bugs Found and Fixed

- **sizeof(ramfb_cfg) = 32 (padded)**: fw_cfg file is exactly 28 bytes.
  QEMU rejects DMA writes with mismatched length. Fix: hardcode length=28.

- **virtio-gpu INVALID_RESOURCE_ID**: On QEMU 10.2, CREATE_2D returns
  RESP_OK_NODATA but ATTACH_BACKING returns 0x1203 (invalid resource).
  Confirmed with hex dumps showing correct wire format. Root cause unknown.
  Workaround: use ramfb instead of virtio-gpu.

- **seL4_TCB_BindNotification + seL4_Call conflict**: Binding a notification
  to a TCB causes seL4_Call (which does Send+Recv) to return the notification
  badge instead of the IPC reply when an IRQ arrives during the call.
  Fix: use seL4_Wait on unbound notification instead.

- **build_sbase.py --clean deletes dash**: The --clean flag removes
  build-04/sbase/ including the dash binary. Dash must be rebuilt AFTER
  sbase --clean. Correct order: ninja -> sbase -> dash -> disk.

- **gpu_cmd poll race**: Must capture used->idx BEFORE kick (avail update +
  QUEUE_NOTIFY), not after. If device processes synchronously, reading
  used->idx after kick sees the post-completion value and poll never exits.

## Lessons Learned

- **fw_cfg MMIO is DEVICE_BIG_ENDIAN**: QEMU byte-swaps writes for
  big-endian devices. Guest must write bswap(value) so QEMU double-swaps
  back to the intended value. All fw_cfg DMA struct fields are also BE.

- **Struct padding breaks wire protocols**: C compilers pad structs to
  alignment of largest member. sizeof(ramfb_cfg) = 32 but wire format is
  28. Always use hardcoded sizes for hardware/protocol data.

- **seL4 notification semantics**: Notifications are one-shot accumulating
  signals. seL4_Wait on an unbound notification works fine. Binding is only
  needed for seL4_Recv-based notification checking (which interferes with
  seL4_Call).

- **Rebuild order matters**: When libaios_posix.a changes, ALL binaries
  must be rebuilt: ninja (CMake apps), sbase (build_sbase.py), dash
  (aios-cc), then disk (mkdisk.py). Order: ninja -> sbase -> dash -> disk.

    -machine virt,virtualization=on \

    -serial mon:stdio \

    -drive file=disk/disk_ext2.img,format=raw,if=none,id=hd0 \
